# Дизайн: единое аналитическое ядро — полезные дэши и расследования

> Второй документ (продолжение `HARNESS_REVIEW.md`). Отвечает на вопрос: **как срастить
> SQL-агент и BI-генератор в одну систему**, которая строит сложные, но полезные дэшборды
> и проводит сложные, но полезные расследования — для бизнес-пользователя, на модели
> `gigachat-3-ultra`. Документ — спецификация для реализации (пишется под implement-агента):
> схемы данных, контракты модулей, формулы, каталоги виджетов и детекторов, LLM-вызовы,
> роадмап с критериями приёмки.

---

## 0. TL;DR

Сейчас в репозитории два продукта, которые не знают друг о друге:

- **SQL-агент** умеет точно отвечать на вопрос, но его ответ — CSV-файл, тупик.
- **Отчёт/расследование** умеют красиво показывать, но работают на слепом дампе всей таблицы
  в pandas и решают «что интересно» без опоры на вопрос пользователя, бизнес-контекст и
  масштаб данных — отсюда «фигня».

Лечение — не склеить их скотчем, а вынести общее в **аналитическое ядро** из четырёх слоёв,
поверх которого и агент, и дэши, и расследования становятся тонкими сценариями:

```
             ┌────────────────────────────────────────────────────┐
             │                 ПОВЕРХНОСТИ (тонкие)                │
             │  /ask (text2sql)   /dash (дэшборд)   /investigate   │
             └─────────┬──────────────┬──────────────┬─────────────┘
                       ▼              ▼              ▼
  ┌────────────────────────────────────────────────────────────────┐
  │ 1. SEMANTIC MODEL   — что такое эти данные (меры, разрезы,     │
  │    сущности, связи, правила скоупа, единицы) — ЕДИНАЯ           │
  │ 2. QUERY ENGINE     — как их достать (AggregateRequest → SQL   │
  │    push-down в Greenplum; pandas только для малых результатов)  │
  │ 3. INSIGHT ENGINE   — что в них важно (детекторы + скоринг     │
  │    полезности + критик) — ЕДИНЫЙ для дэшей и расследований      │
  │ 4. RENDER ENGINE    — как показать (виджеты: Plotly.js графики │
  │    + Tabulator таблицы → один офлайн-HTML)                      │
  └────────────────────────────────────────────────────────────────┘
```

Ключевая идея против «фигни»: **польза формализуется и измеряется** (§2, §6), а не
делегируется вкусу LLM. LLM планирует и рассказывает; отбирает — скоринг и критик.

## 1. Диагноз: почему сейчас получается фигня

Честный разбор механики текущего кода, из-за которой отчёты периодически бесполезны:

**Д-1. Нет вопроса — нет пользы.** `/build_business_report table` без фокуса запускает
«майнинг всего по всему»: 10 разрезов × все меры × 4 типа детекторов. Что выжило по
статистическим порогам — то и в отчёт. Но статистическая выраженность ≠ бизнес-значимость:
«доля флага X по коду источника отклоняется в 1.4×» — правда, и что? Отчёт без адресата
и вопроса — это распечатка EDA, а не аналитика.

**Д-2. LLM решает «что интересно» вслепую.** Роли колонок и производные показатели LLM
назначает по именам и 5 примерам значений — не видя распределений, не зная бизнес-приоритетов,
без права на «эта таблица скучная, вот единственные 2 вещи, которые стоит показать».
При этом отбор находок (`_select`) — чисто числовой, без семантики: тривиальное («у 96%
задач нет просрочки — отклонение!») проходит, важное может отсеяться капом на тип.

**Д-3. Одна таблица = потолок глубины.** Самые ценные инсайты живут на стыках: отток
(факт) × сегмент (справочник ЕПК) × зарплатный проект (payroll). Отчёт принципиально
одно-табличный, хотя join-граф с проверенными связями уже есть в каталоге — им пользуется
только SQL-агент.

**Д-4. Пороги от балды, калибровки нет.** 0.05 / 1.3 / 1.8 / 25% / 4× — рассыпаны по коду,
подобраны на четырёх демо-таблицах. На другой таблице те же пороги дают либо пустой отчёт,
либо стену мусора. Нет ни конфига чувствительности, ни адаптивных порогов от объёма данных.

**Д-5. Нет проверки «и что с того?».** Ни одна находка не обязана отвечать на вопрос
«какое решение может принять человек, увидев это?». Нет ни машинного критика, ни правила
композиции «максимум N секций, отранжированных по денежному эффекту».

**Д-6. Нет обратной связи.** Пользователь не может сказать «эта секция — мусор», и система
не может перестать генерировать такие секции. При фиксированной модели фидбек — единственный
механизм роста качества.

Каждому пункту ниже соответствует конкретный механизм: Д-1→§5.2 (вопрос обязателен, режимы),
Д-2→§4+§6.2 (семантическая модель + скоринг), Д-3→§4.3+§5.4 (граф связей в ядре),
Д-4→§6.3 (профили чувствительности, адаптивные пороги), Д-5→§6.4 (критик + композиция),
Д-6→§9 (фидбек-цикл).

## 2. Что такое «полезно»: формализация

Прежде чем строить, фиксируем определение. **Виджет/находка полезны, если бизнес-пользователь
может назвать действие или решение, которое из них следует.** Операционализируем четырьмя
осями (они станут скорингом в §6.2):

| Ось | Вопрос | Пример «да» | Пример «нет» |
|---|---|---|---|
| **Материальность** | Затрагивает ли значимую долю денег/людей/объёма? | сегмент даёт 40% потерь ФОТ | срез из 12 записей (0.1%) |
| **Неожиданность** | Отличается ли от базы/ожидания/прошлого периода? | просрочка ×2.3 к среднему | «в крупном сегменте крупные суммы» (тавтология) |
| **Адресность** | Понятно ли, КТО/ГДЕ именно? (имя, территория, менеджер) | «ГОСБ 38, топ-5 компаний = 60%» | «в среднем по таблице» |
| **Действенность** | Есть ли ручка? (вернуть, проверить, перераспределить) | компании с высоким потенциалом среди ушедших | распределение системного кода загрузки |

Плюс два **жёстких стоп-правила** (нарушение = виджет не попадает в выдачу):

- **Стоп «тавтология»**: находка не должна следовать из определения данных. Детектируется
  правилами: доля флага ~0% или ~100% при null-семантике; корреляция меры с самой собой
  через производную; лидерство самой крупной категории по абсолютной сумме без нормировки.
- **Стоп «нечитаемо»**: разрез с кардинальностью, не влезающей в виз (>15 категорий без
  «прочие»), id без человеческой подписи, единица измерения не определена.

Это определение — контракт всей системы: скоринг (§6.2), критик (§6.4), eval (§10) и
фидбек (§9) измеряют одно и то же.

## 3. Целевая архитектура

### 3.1 Слои и их обязанности

```
src/text2sql/
  semantics/        # СЛОЙ 1: семантическая модель
    model.py        #   pydantic-схемы SemanticTable/Measure/Dimension/JoinPath
    store.py        #   SQLite-хранилище + загрузка/сохранение + версии
    enrich.py       #   построение модели: интроспекция+статистика+ОДИН LLM-вызов+правки человека
  engine/           # СЛОЙ 2: query engine
    request.py      #   AggregateRequest / QueryResultFrame (контракты)
    compile.py      #   AggregateRequest → SQL (sqlglot AST, диалект, dedup, скоуп-правила)
    executor.py     #   исполнение: батч запросов, кэш, лимиты, оценка размера
  insight/          # СЛОЙ 3: insight engine
    detectors.py    #   каталог детекторов (работают на РЕЗУЛЬТАТАХ агрегатов, не на сырье)
    scoring.py      #   формула полезности, профили чувствительности, стоп-правила
    critic.py       #   LLM-критик (отбор/отбраковка) + правила композиции
  render/           # СЛОЙ 4: рендер
    widgets.py      #   WidgetSpec → фрагмент HTML/JS (Plotly) | таблица (Tabulator)
    page.py         #   сборка дэша: layout, тема, офлайн-инлайн ассетов, TOC, фидбек-кнопки
    assets/         #   vendored: plotly.min.js, tabulator.min.js+css (офлайн!)
  surfaces/         # ПОВЕРХНОСТИ
    ask.py          #   text2sql-агент (текущий graph/, переведён на engine/)
    dash.py         #   /dash — вопрос → DashboardSpec → исполнение → HTML
    investigate.py  #   /investigate — рекурсивная декомпозиция на engine/
```

### 3.2 Главный принцип срастания

**И дэш, и расследование, и SQL-агент говорят с данными через один язык —
`AggregateRequest`** (§5.1). SQL-агент компилирует в него `StructuredPlan`; дэш — виджеты;
расследование — шаги декомпозиции. Один компилятор SQL, один кэш, один анти-fanout, одни
скоуп-правила. Отчёт перестаёт качать таблицу в pandas — Greenplum сам считает агрегаты
(эти запросы он выполняет за секунды на любом объёме).

Побочный эффект: **все текущие анти-fanout наработки (check_join, dedup, join-граф)
автоматически начинают защищать и отчёты**, которые сейчас вообще не джойнят.

## 4. Слой 1: семантическая модель

### 4.1 Схема (pydantic, хранение в SQLite `data_for_agent/semantics.db`)

```python
class Unit(str, Enum): money="money"; people="people"; count="count"; percent="percent"; days="days"; other="other"

class Measure(BaseModel):
    name: str                  # тех.имя колонки ИЛИ производной
    label: str                 # «Отток ФОТ»
    expr: str                  # SQL-выражение: "outflow_fot_amt" | "CASE WHEN fact>plan THEN 1 ELSE 0 END"
    agg: Literal["sum","avg","count","count_distinct"]
    unit: Unit
    good_direction: Literal["up","down","none"]   # рост — это хорошо или плохо (для окраски и формулировок!)
    priority: int = 0          # 0=главная мера таблицы … (LLM предлагает, человек правит)
    scope_sql: str | None      # "task_category='Предложение'" — где мера осмысленна (из scope_rules)

class Dimension(BaseModel):
    name: str; label: str
    card: int                  # кардинальность (из статистики)
    kind: Literal["category","geo","org","reason","status","channel","source"]
    id_partner: str | None     # gosb_id ↔ gosb_name
    top_values: list[str]      # до 25 значений для резолва фильтров

class Entity(BaseModel):
    name: str; label: str      # «Компания»
    key_col: str               # inn
    name_col: str | None       # company_name
    value_col: str | None      # колонка «ценности» (потенциал/ФОТ) для приоритизации

class TimeCol(BaseModel):
    name: str; label: str
    is_reporting: bool         # отчётная дата (ось анализа) vs событийная
    granularity: Literal["day","month"]         # определяется по данным
    fresh_until: date | None   # последний ПОЛНЫЙ период (детект неполного месяца!)

class JoinPath(BaseModel):
    to_table: str
    on: list[tuple[str,str]]
    cardinality: Literal["N:1","1:1"]           # только безопасные пути попадают сюда
    verified_at: date          # когда check_join подтвердил живой пробой
    enrich_cols: list[str]     # что полезно тащить: ["gosb_name","segment_name"]

class SemanticTable(BaseModel):
    fqn: str; label: str; grain: str; row_meaning: str      # «одна строка = задача воронки»
    audience_hint: str         # «руководителям сети продаж» — контекст для планировщика/нарратора
    time: list[TimeCol]; measures: list[Measure]; dimensions: list[Dimension]
    entities: list[Entity]; joins: list[JoinPath]
    boring_cols: list[str]     # явный список «не анализировать» (тех.поля) — вместо угадываний
    question_bank: list[str]   # 5-10 вопросов, на которые эта таблица отвечает (см. §4.2)
```

### 4.2 Как модель строится (enrich.py)

При `/refresh_metadata` для каждой таблицы:

1. **Детерминированная часть** (без LLM): интроспекция, статистика по адаптивному сэмплу
   (кардинальности, доли null/нулей, min/max дат, доля дробных), автоопределение
   `granularity`, `fresh_until`, `id_partner`-пар, `card`.
2. **Один структурный LLM-вызов** «семантическое профилирование» — вход: имена+описания+
   статистика+примеры; выход: labels, units, kinds, grain, row_meaning, приоритеты мер,
   производные (overdue/duration/rate/impact — текущий `_BEHAV_SYS` вливается сюда),
   scope-правила, `good_direction`, `boring_cols` и **question_bank** — «на какие 5–10
   бизнес-вопросов отвечает таблица». Question bank — ключ к пользе: планировщик дэша
   (§5.2) обязан привязывать каждый виджет к вопросу из банка или из запроса пользователя.
3. **Правки человека** — интерактив в CLI (замена нынешнего `money_confirm`): показать
   модель, дать поправить единицы/приоритеты/скучные колонки. Правки сохраняются с
   `source="human"` и **никогда не перетираются** LLM-ом при следующем рефреше.
4. **Join-пути**: из join-графа берутся только пары, прошедшие `check_join` (N:1/1:1),
   с записью `verified_at`; для справочной стороны фиксируются `enrich_cols`.

### 4.3 Кросс-табличность

Семантическая модель хранит связи → engine умеет **обогащение**: любой `AggregateRequest`
к факту может группироваться по колонке из справочника (`dim: "uzp_dim_gosb.gosb_name"`),
компилятор сам подставит верифицированный join (+dedup при необходимости). Это открывает
дэшам и расследованиям стыковые инсайты (Д-3) без каких-либо усилий на поверхности.

## 5. Слой 2: Query Engine и Слой поверх него — дэшборды

### 5.1 Контракт `AggregateRequest`

Единственный способ для всех поверхностей получить данные:

```python
class AggregateRequest(BaseModel):
    table: str
    measures: list[str]                 # имена мер из семантической модели
    dims: list[str] = []                # "segment_name" | "uzp_dim_gosb.gosb_name" | "time:month"
    filters: list[Filter] = []          # переиспользуем Filter из plan.model
    top_n: int | None = None            # + автоматическое «прочие» (сервер считает остаток!)
    order_by_measure: str | None = None
    compare: Compare | None = None      # см. ниже
    include_share: bool = True          # долю от итога считает SQL (SUM() OVER ())

class Compare(BaseModel):               # декларативное сравнение периодов — в ОДНОМ запросе
    time_col: str
    kind: Literal["prev_period","same_period_last_year","period_over_period_series"]
```

Компилятор (`compile.py`, на **sqlglot**):
- строит GROUP BY с `top_n`+«прочие» (оконный ранг), долями, compare (условные агрегаты
  по двум периодам) — всё в одном запросе;
- применяет `scope_sql` меры автоматически (Д-«потенциал только для Предложений» решён раз
  и навсегда на уровне движка);
- применяет dedup/join из семантической модели;
- транспилирует под диалект, оценивает cost, ставит probe-таймаут.

Executor: батч независимых запросов (дэш из 8 виджетов = 8 GROUP BY — пускать
последовательно, GP быстрый; параллельность — v2), кэш результатов по хэшу запроса
(один дэш часто переиспользует агрегаты), потолок на размер результата (top_n обязателен
для дэшей).

`QueryResultFrame` = маленький pandas.DataFrame результата + метаданные (единицы, доли,
compare-колонки) — на нём работают детекторы и рендер.

### 5.2 `/dash` — от вопроса к дэшборду

Команда: `/dash <таблица|тема> <вопрос или задача> [--where ...]`.
**Вопрос обязателен** (Д-1). Если пользователь дал только таблицу — система не «майнит всё»,
а предлагает выбрать из `question_bank` таблицы (нумерованный список) или ввести свой.
Режим «полный обзор» остаётся, но как явный выбор `[0] Общий обзор таблицы`, и даже он
планируется через question bank, а не через «всё по всему».

Конвейер:

```
вопрос ─► ПЛАНИРОВЩИК (LLM-вызов №1, структурный) ─► DashboardSpec (черновик)
      семантическая модель как контекст            │
                                                    ▼
                            ВАЛИДАЦИЯ + ОБОГАЩЕНИЕ (код): имена существуют, кардинальности
                            влазят в виз, добавить обязательные виджеты (KPI-шапка, динамика
                            главной меры), выкинуть дубли
                                                    │
                                                    ▼
                            ИСПОЛНЕНИЕ (engine, батч) → QueryResultFrame'ы
                                                    │
                                                    ▼
                            INSIGHT ENGINE (§6): детекторы на результатах, скоринг,
                            стоп-правила, ранжирование виджетов по score
                                                    │
                                                    ▼
                            КРИТИК (LLM-вызов №2): отбраковка тривиального, заголовки-выводы
                                                    │
                                                    ▼
                            НАРРАТОР (LLM-вызов №3): «Главное», выводы секций, «что делать»
                                                    │
                                                    ▼
                            RENDER: интерактивный HTML (Plotly+Tabulator), MD-версия
```

Итого **3 LLM-вызова на дэш** (сейчас 5–6), все структурные.

### 5.3 `DashboardSpec` — декларативный контракт (то, что производит LLM-планировщик)

```python
class WidgetSpec(BaseModel):
    id: str
    question: str               # ← ОБЯЗАТЕЛЬНО: на какой вопрос отвечает виджет (Д-5)
    viz: Literal["kpi_row","bar","stacked_bar","line","area","pareto","heatmap",
                 "treemap","waterfall","scatter_quadrant","table","funnel"]
    request: AggregateRequest   # данные виджета
    baseline: Literal["none","overall_avg","prev_period"] = "none"   # против чего сравниваем
    size: Literal["full","half"] = "half"

class DashboardSpec(BaseModel):
    title: str
    audience: str               # для тона нарратива
    question: str               # исходный вопрос пользователя (переформулированный)
    filters_global: list[Filter]
    widgets: list[WidgetSpec]   # черновик 8-14, после критика останется 5-9
```

Планировщику в промпт идёт **компактная семантическая модель** (labels, меры с приоритетами
и единицами, разрезы с кардинальностями, сущности, join-пути, question_bank, fresh_until) +
вопрос + 2–3 примера хороших DashboardSpec (few-shots из золотого набора, §9). Ему **не**
идут сырые данные — план строится по семантике, проверяется исполнением.

### 5.4 Каталог виджетов (render/widgets.py)

Каждый тип — детерминированная функция `(WidgetSpec, QueryResultFrame, InsightCard|None) → WidgetHTML`:

| viz | Когда | Реализация |
|---|---|---|
| `kpi_row` | шапка дэша | карточки: значение, Δ к пред.периоду (стрелка, окраска по `good_direction`!), спарклайн (Plotly) |
| `bar` | мера по разрезу ≤15 категорий | горизонтальный бар, пунктир базы, числа на барах, тултип с долей и n |
| `line`/`area` | динамика | месяцы, маркеры аномалий из детектора, серая зона «неполный период» (fresh_until!) |
| `pareto` | концентрация | бары+кумулята, подсветка «сколько категорий = 80%» |
| `stacked_bar` | структура во времени | доли слагаемых по месяцам (компоненты target) |
| `heatmap` | 2 разреза × мера | Plotly heatmap, аннотации, ≤8×8 + «прочие» |
| `treemap` | иерархия (разрез→сущность) | **нативный Plotly treemap** (интерактивный drill кликом — заменяет самодельный squarify) |
| `waterfall` | как складывается итог | нативный Plotly waterfall |
| `scatter_quadrant` | приоритизация (убыль×ценность) | подписи топ-точек, медианные линии |
| `table` | детальные списки («кто») | **Tabulator**: сортировка, поиск, фильтры по колонкам, закреплённая шапка, формат чисел/₽, цветовые бары в ячейках, экспорт CSV кнопкой |

Правила зашиты в рендер (не в LLM): деньги — с ₽ и разрядными пробелами; отрицательное
«хорошее» — зелёным по `good_direction`; >12 категорий — «прочие»; каждая ось подписана
label-ом из семантической модели.

### 5.5 Интерактивный HTML (то, что «понравилось, но не запомнил»)

- **Plotly.js** — все графики: зум, пан, тултипы с точными значениями, вкл/выкл серий
  легендой, встроенный экспорт PNG. Treemap/waterfall/funnel — нативные, интерактивный
  drill-down в treemap кликом бесплатно.
- **Tabulator** (MIT) — все таблицы: клиентская сортировка/фильтрация/поиск, пагинация,
  группировка строк, форматтеры (деньги, проценты, прогресс-бары в ячейках), экспорт CSV.
- **Офлайн-режим обязателен** (банковский контур без интернета): `plotly.min.js` (~3.5 МБ)
  и `tabulator.min.js+css` кладутся в пакет (`render/assets/`) и **инлайнятся** в HTML.
  Итоговый файл 4–6 МБ — приемлемо; данные виджетов — inline JSON (`<script type=
  "application/json">`), не base64-картинки.
- Каркас страницы: шапка (вопрос, фильтры, период данных, предупреждение о неполном месяце),
  блок «🎯 Главное» с 3–5 пунктами-ссылками на виджеты, сетка виджетов (CSS grid,
  full/half), у каждого виджета: заголовок-вывод (не «ТОП по X», а «60% потерь — три
  ГОСБа»), бейдж score, кнопки 👍/👎 (§9), сворачиваемый блок «как посчитано» (SQL запроса —
  доверие через прозрачность).
- Лёгкий cross-filter (v2): клик по категории бара подсвечивает эту категорию в остальных
  виджетах (данные уже на клиенте; без пересчёта агрегатов).
- MD-версия сохраняется параллельно (для почты/вики) со статичными PNG (kaleido или
  текущий matplotlib-путь).

## 6. Слой 3: Insight Engine — гарантия «не фигни»

### 6.1 Детекторы работают на агрегатах

Все текущие детекторы (`rate_dev`, `money_conc`, `value_mismatch`, `interaction`,
`spikes`, `seasonality`, `repeat`) переписываются с сигнатуры `(df_raw, ...)` на
`(QueryResultFrame, context)` — вход уже агрегирован движком. Добавляются:

- `trend_break` — перелом тренда (последние 2–3 точки против экстраполяции предыдущих);
- `new_leader` — смена лидера разреза между периодами (compare в одном запросе);
- `share_shift` — структурный сдвиг долей (сегмент вырос с 20%→35% в потерях);
- `benchmark_gap` — срез против медианы срезов той же меры («ГОСБ 38 хуже 80% ГОСБов»);
- `data_quality` — предпроверки: константные колонки, разрез=1 значение, мера=все нули,
  неполный последний период → эти сигналы не в отчёт, а в **подавление** виджетов и в
  плашку «период неполный».

Каждый детектор возвращает `InsightCard`:

```python
class InsightCard(BaseModel):
    widget_id: str
    kind: str                          # тип детектора
    headline_facts: dict               # числа для заголовка/нарратива (как сейчас facts)
    materiality: float                 # 0..1  — доля затронутых денег/объёма от целого
    surprise: float                    # 0..1  — нормированное отклонение (lift/z по рангу)
    addressability: float              # 0..1  — есть имя/территория/сущность? глубина адреса
    actionability: float               # 0..1  — по типу: value_mismatch=0.9, quality=0.2 …
    score: float                       # см. 6.2
    stop_flags: list[str]              # «тавтология», «нечитаемо», «дубль» → не показывать
```

### 6.2 Формула полезности (единая, конфигурируемая)

```
score = (w_m·materiality + w_s·surprise + w_a·addressability + w_x·actionability)
        · money_boost      # ×1.5 если мера unit=money
        · focus_boost      # ×1.3 если виджет напрямую отвечает на вопрос пользователя
        · penalty_dup      # ×0.5 за каждый повтор того же значения среза в уже отобранных
```

Веса и пороги — в **одном** конфиге `insight/scoring.yaml` с тремя профилями
чувствительности: `strict` (только громкое, для руководителя), `default`, `explore`
(для аналитика). Пороги материальности — **адаптивные**: `min_rows = max(30, 0.5% строк)`,
`min_money_share = 3%` (не абсолютные числа). Это закрывает Д-4.

### 6.3 Стоп-правила (детерминированные, до критика)

- тавтология: доля ≥97%/≤3% у rate-меры без денежного веса; лидер по сумме = лидер по
  количеству при сопоставимых средних («большой сегмент большой»); мера сравнивается со
  своей же производной;
- нечитаемость: нет label/unit; кардинальность не влазит; «прочие» >70% (разрез не режет);
- дубль: та же пара (мера, значение среза) уже показана виджетом с большим score;
- неполный период: находка «падение в последнем месяце», если `fresh_until` говорит, что
  месяц не закрыт → находка подавляется, вместо неё плашка.

### 6.4 LLM-критик (вызов №2)

Вход: список карточек (заголовочные факты + question виджета + score), вопрос пользователя,
аудитория. Выход (структурный):

```json
{"verdicts": [{"id":"w3","keep":true,"headline":"Три ГОСБа дают 61% потерь ФОТ",
               "reason_if_drop":null}, ...],
 "ordering": ["w1","w3","w7", ...],
 "missing": ["нет разбивки по времени — когда началось?"]}
```

Правила критику: держать 5–9 виджетов; отбраковывать тавтологии, которые проскочили правила;
у каждого выжившего — заголовок-вывод с числом (не название среза). Поле `missing` —
если критик находит дыру, и она выражается одним `AggregateRequest` — движок дозапрашивает
и виджет добавляется (одна итерация, не цикл). Это самовосполнение — дешёвая версия
«агентности» без бесконечных циклов.

### 6.5 Нарратор (вызов №3)

Как текущий `narrate`, но: (а) получает только выживших после критика; (б) получает
**связи** — предвычисленный список «значение X встречается в виджетах w1,w3,w5» (сквозной
сюжет: «ГОСБ 38 фигурирует и в потерях, и в просрочке — вероятно, системная проблема
территории»); (в) обязан в «Главном» дать 3–5 пунктов вида «факт с числом → значит →
что сделать»; (г) запрет матжаргона — как сейчас.

## 7. Расследования 2.0

Расследование = **направленный обход того же ядра**. Отличие от дэша: дэш — ширина
(панорама по вопросу), расследование — глубина (цепочка «почему» до конкретных имён).

### 7.1 Конвейер

```
вопрос ─► FRAME (LLM №1): target-мера, направление (loss/gain/top), кандидаты-разрезы,
          кандидаты-причины, сущность, мера ценности — ВСЁ из семантической модели
              │
              ▼
   РЕКУРСИВНЫЙ СПУСК (код, без LLM): на каждом уровне
     • для всех разрезов-кандидатов: AggregateRequest(top_n=12) → декомпозиция вклада
     • выбираем разрез с максимальной концентрацией (как сейчас top_share)
     • фиксируем уровень, спускаемся в топ-1..2 узла (фильтр), убираем использованный разрез
     • стоп: глубина 4 / вклад узла <15% / < min_rows → честное «дальше размазано»
     ГЛАВНОЕ ОТЛИЧИЕ: спуск может ПЕРЕСЕКАТЬ таблицы по join-путям семантической модели
     (сегмент из ЕПК, названия ГОСБ из справочника) — стыковые расследования (Д-3)
              │
              ▼
   ВРЕМЯ (код): динамика target по месяцам в разрезе найденного «горячего» пути;
   trend_break → «когда началось»; сравнение до/после точки перелома
              │
              ▼
   ПОЧЕМУ (код): разрезы-причины С НОРМИРОВКОЙ (lift против базового распределения:
   «в горячем срезе причина Х встречается ×2.4 чаще, чем в среднем» — а не «преобладает
   Иное (80%)», которое преобладает везде)
              │
              ▼
   КТО (код): сущности внутри горячего пути + ценность + Парето + квадрант приоритета
              │
              ▼
   ГИПОТЕЗЫ-ПРОВЕРКИ (LLM №2, опционально): по собранным фактам LLM предлагает ≤3
   проверяемых гипотезы в виде AggregateRequest («если это сезонность — тот же месяц
   прошлого года покажет то же»); движок исполняет; результат = подтверждено/опровергнуто
              │
              ▼
   СИНТЕЗ (LLM №3): ответ, причинная цепочка с числами, действия, ЯВНЫЙ блок
   «что НЕ подтвердилось» (опровергнутые гипотезы — это тоже ценность и честность)
```

### 7.2 Выход расследования

Тот же render-движок: интерактивный HTML, где дерево спуска — **Plotly treemap/icicle
с реальным drill-down по клику** (иерархия уровней уже посчитана), временной график с
маркером перелома, таблица «кто» — Tabulator с сортировкой по ценности, waterfall итога.
Структура: Ответ → Когда началось → Где сосредоточено (дерево) → Почему (lift) →
Кто (+приоритет) → Проверенные гипотезы → Что делать.

### 7.3 Смычка поверхностей (сценарии, которые становятся возможными)

1. `/ask` → результат → кнопка/команда **«дэш по этому»**: одобренный план агента — это
   готовые table+filters для `DashboardSpec.filters_global`.
2. Виджет дэша → «расследовать» → `/investigate` стартует с target и фильтрами виджета
   (не с нуля).
3. Расследование само генерирует `--where` из вопроса («к 2026» → фильтр по времени)
   через тот же резолв фильтров, что у агента.

## 8. LLM-оркестрация под gigachat-3-ultra

Бюджет вызовов (все — структурный вывод через function calling, см. HARNESS_REVIEW §6.4):

| Сценарий | Вызовы | Комментарий |
|---|---|---|
| refresh таблицы | 1 | семантическое профилирование (кэшируется в SQLite навсегда до re-refresh) |
| /dash | 3 (план, критик, нарратор) | +1 опц. на «missing»; при 5с rate-limit ≈ 15–20 с LLM-времени |
| /investigate | 2–3 (frame, [гипотезы], синтез) | спуск/время/почему/кто — 0 вызовов |
| /ask | 2–3 | как в HARNESS_REVIEW §6.2 |

Принципы: (1) LLM никогда не считает числа — только выбирает и формулирует; (2) каждый
вызов имеет pydantic-схему и валидацию с одним ретраем; (3) few-shots подтягиваются из
золотого набора (§9) по похожести вопроса; (4) семантическая модель в промпте — компактный
рендер (label|unit|priority|card), не сырой JSON.

## 9. Обратная связь: как система перестаёт делать фигню со временем

1. **Кнопки 👍/👎 на каждом виджете** в HTML пишут событие в `feedback.jsonl` через
   `fetch` на локальный эндпоинт... в офлайн-HTML это невозможно — поэтому v1: кнопка
   копирует в буфер команду `/feedback w3 down "почему"`, пользователь вставляет в CLI
   (одно движение); v2 (web-UI): честные кнопки.
2. Фидбек хранится структурно: `{table, question, widget_spec, insight_kind, verdict, reason}`.
3. Использование: (а) few-shots критику — примеры «это пользователи считают мусором»;
   (б) понижающие коэффициенты в скоринге на пары (детектор × тип разреза), стабильно
   получающие 👎; (в) одобренные дэши целиком (`/dash_save`) → золотой набор few-shots
   планировщику.
4. **Золотой набор** — 5–10 вручную выверенных `DashboardSpec` + расследований на реальных
   таблицах — создаётся при внедрении и является одновременно few-shots и eval-фикстурами.

## 10. Eval полезности (иначе «стало лучше» неизмеримо)

1. **Структурные проверки** (без LLM): каждый виджет привязан к question; нет стоп-флагов
   в выдаче; заголовки содержат числа; единицы у всех мер; ≤9 виджетов; KPI-числа дэша
   сходятся с прямым SQL-запросом (**числовая корректность** — сейчас не проверяется вообще).
2. **LLM-судья по рубрике §2** (осознавая иронию: gigachat судит gigachat — но для
   регрессионного сравнения A/B версий годится): оценка каждой секции по 4 осям 0–2,
   агрегат по дэшу. Прогон на золотом наборе → тренд в CI.
3. **Анти-фигня-кейсы**: специально скучные таблицы (все меры-нули, один сегмент,
   константы) — система обязана выдать короткий честный дэш «показывать почти нечего,
   вот 2 факта» вместо стены шума. Это отдельный класс приёмочных тестов.
4. Кейсы расследований с **известной подложенной причиной** (сидер уже умеет: «горячий
   срез просрочки», «перекос деньги/количество») — расследование обязано её найти;
   метрика recall@найденная-причина.

## 11. План реализации (для implement-агента)

### Фаза 1 — фундамент (ядро без смены UX)
1. `semantics/`: схемы, SQLite-store, enrich с одним LLM-вызовом, миграция из текущих CSV
   (конвертер), интерактивная правка человека. Текущие labels/roles/behaviors-вызовы
   удаляются, их потребители читают модель.
2. `engine/`: AggregateRequest, компилятор на sqlglot (top_n+прочие, доли, compare,
   scope_sql, dedup, join-пути), executor с кэшем и лимитами.
3. `render/`: vendored Plotly+Tabulator, каталог виджетов (bar, line, kpi_row, table,
   pareto, heatmap, treemap, waterfall, scatter_quadrant), сборка страницы, офлайн-инлайн.
   Приёмка: текущий отчёт собирается на новом рендере (даже со старым майнингом).

### Фаза 2 — /dash (замена /build_business_report)
4. DashboardSpec + планировщик (LLM №1) + валидация/обогащение.
5. Insight engine: детекторы на агрегатах, scoring.yaml с профилями, стоп-правила.
6. Критик + нарратор + композиция. `/build_business_report` становится алиасом
   `/dash <table> "общий обзор"` (обратная совместимость).
   Приёмка: eval §10.1+10.3 зелёный; на 6 демо-таблицах дэши ≤9 виджетов, каждый с
   вопросом и заголовком-выводом; KPI сходятся с SQL.

### Фаза 3 — /investigate 2.0
7. Рекурсивный спуск на engine (в т.ч. кросс-табличный), время+trend_break, lift-почему.
8. Гипотезы-проверки, синтез с «что не подтвердилось», treemap с drill.
   Приёмка: §10.4 — подложенные причины находятся 3/3; кросс-табличный кейс
   (отток×сегмент из ЕПК) работает.

### Фаза 4 — смычка и фидбек
9. «Дэш по результату /ask», «расследовать виджет», /feedback + золотой набор +
   few-shots-подтяжка, LLM-судья в eval.

### Определение успеха (продуктовое)
Бизнес-пользователь на **незнакомой системе таблице** задаёт свой вопрос через `/dash`
и получает за ~1 минуту страницу, где: каждый блок отвечает на понятный вопрос; главные
выводы названы с числами и именами; есть минимум одно действие, которое он готов забрать
в работу; и ни одного блока, про который он спросит «зачем мне это?». Это и есть критерий,
по которому проверяем каждую фазу на живых пользователях, а не только eval-ом.
