# Дизайн: единое аналитическое ядро — полезные дэши и расследования

> Второй документ (продолжение `HARNESS_REVIEW.md`). Отвечает на вопрос: **как срастить
> SQL-агент и BI-генератор в одну систему**, которая строит сложные, но полезные дэшборды,
> проводит сложные, но полезные расследования и умеет **доказывать эффективность
> инструментов** (impact-анализ) — для бизнес-пользователя, на модели `gigachat-3-ultra`.
> Документ — спецификация для реализации: схемы данных, контракты модулей, формулы,
> каталоги виджетов и детекторов, LLM-вызовы, роадмап с критериями приёмки.
>
> **Разделение документов:** `HARNESS_REVIEW.md` — аудит самого агента (граф, LLM/DB-слои,
> каталог, CLI, платформа); этот документ — всё про аналитические функции. Находки аудита
> по отчётам/расследованиям перенесены сюда из HARNESS_REVIEW с сохранением номеров B-* —
> см. приложение §13, где каждая привязана к разделу дизайна, который её решает.
>
> **Зафиксированные решения владельца** (v2 документа):
> 1. **Никакого SQLite** — файловые локи SQLite виснут из-за Jupyter-ядер (мёртвое ядро
>    держит соединение). Всё персистентное — **JSON-файлы** с атомарной записью.
> 2. **Вычисления — pandas, не БД.** На машине 120 ГБ RAM; БД (Greenplum) нельзя класть
>    аналитическими вычислениями. БД используется ровно один раз на сценарий — отдать
>    данные; всё остальное считает pandas в памяти.
> 3. Новый обязательный сценарий: **режим «Эффективность инструмента»** — по признаку
>    вида `is_in_pipeline` система сама находит, где инструмент используется, выделяет
>    пилотную группу, честно сравнивает исходы и экстраполирует эффект (§8).

## 0. TL;DR

Сейчас в репозитории два продукта, которые не знают друг о друге:

- **SQL-агент** умеет точно отвечать на вопрос, но его ответ — CSV-файл, тупик.
- **Отчёт/расследование** умеют красиво показывать, но решают «что интересно» без опоры
  на вопрос пользователя, бизнес-контекст и семантику данных — отсюда «фигня».

Лечение — не склеить их скотчем, а вынести общее в **аналитическое ядро** из четырёх
слоёв, поверх которого агент, дэши, расследования и impact-анализ становятся тонкими
сценариями:

```
        ┌───────────────────────────────────────────────────────────┐
        │                   ПОВЕРХНОСТИ (тонкие)                     │
        │  /ask (text2sql)  /dash  /investigate  /impact (новое)     │
        └───────┬───────────┬──────────┬───────────┬─────────────────┘
                ▼           ▼          ▼           ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │ 1. SEMANTIC MODEL — что такое эти данные (меры, разрезы,        │
  │    сущности, связи, правила скоупа, единицы). JSON-файлы.        │
  │ 2. DATA ENGINE    — как их достать и посчитать: ОДНА загрузка   │
  │    из БД на сценарий → DataFrame Store в памяти →                │
  │    AggregateRequest исполняется PANDAS-ом (БД не трогаем)        │
  │ 3. INSIGHT ENGINE — что в них важно (детекторы + скоринг        │
  │    полезности + LLM-критик) — единый для всех поверхностей       │
  │ 4. RENDER ENGINE  — как показать (Plotly.js графики +           │
  │    Tabulator таблицы → один офлайн-HTML)                         │
  └─────────────────────────────────────────────────────────────────┘
```

Ключевая идея против «фигни»: **польза формализуется и измеряется** (§2, §6), а не
делегируется вкусу LLM. LLM планирует и рассказывает; отбирает — скоринг и критик;
числа считает pandas детерминированно.

## 1. Диагноз: почему сейчас получается фигня

Честный разбор механики текущего кода, из-за которой отчёты периодически бесполезны:

**Д-1. Нет вопроса — нет пользы.** `/build_business_report table` без фокуса запускает
«майнинг всего по всему»: 10 разрезов × все меры × 4 типа детекторов. Что выжило по
статистическим порогам — то и в отчёт. Но статистическая выраженность ≠ бизнес-значимость:
«доля флага X по коду источника отклоняется в 1.4×» — правда, и что? Отчёт без адресата
и вопроса — это распечатка EDA, а не аналитика.

**Д-2. LLM решает «что интересно» вслепую.** Роли колонок и производные показатели LLM
назначает по именам и 5 примерам значений — не видя распределений, не зная бизнес-приоритетов,
без права сказать «эта таблица скучная, вот единственные 2 вещи, которые стоит показать».
При этом отбор находок (`_select`) — чисто числовой, без семантики: тривиальное («у 96%
задач нет просрочки — отклонение!») проходит, важное может отсеяться капом на тип.

**Д-3. Одна таблица = потолок глубины.** Самые ценные инсайты живут на стыках: отток
(факт) × сегмент (справочник ЕПК) × зарплатный проект (payroll). Отчёт принципиально
одно-табличный, хотя join-граф с проверенными связями уже есть в каталоге — им пользуется
только SQL-агент.

**Д-4. Пороги от балды, калибровки нет.** 0.05 / 1.3 / 1.8 / 25% / 4× — рассыпаны по коду,
подобраны на четырёх демо-таблицах. На другой таблице те же пороги дают либо пустой отчёт,
либо стену мусора. Нет ни конфига чувствительности, ни адаптивных порогов от объёма данных.

**Д-5. Нет проверки «и что с того?».** Ни одна находка не обязана отвечать на вопрос
«какое решение может принять человек, увидев это?». Нет ни машинного критика, ни правила
композиции «максимум N секций, отранжированных по денежному эффекту».

**Д-6. Нет обратной связи.** Пользователь не может сказать «эта секция — мусор», и система
не может перестать генерировать такие секции. При фиксированной модели фидбек — единственный
механизм роста качества.

Соответствие механизмов: Д-1→§5.3 (вопрос обязателен, режимы), Д-2→§4+§6.2 (семантическая
модель + скоринг), Д-3→§4.3+§5.2 (граф связей + pandas-join в ядре), Д-4→§6.3 (профили
чувствительности, адаптивные пороги), Д-5→§6.4 (критик + композиция), Д-6→§9 (фидбек-цикл).

## 2. Что такое «полезно»: формализация

**Виджет/находка полезны, если бизнес-пользователь может назвать действие или решение,
которое из них следует.** Операционализируем четырьмя осями (они станут скорингом в §6.2):

| Ось | Вопрос | Пример «да» | Пример «нет» |
|---|---|---|---|
| **Материальность** | Затрагивает ли значимую долю денег/людей/объёма? | сегмент даёт 40% потерь ФОТ | срез из 12 записей (0.1%) |
| **Неожиданность** | Отличается ли от базы/ожидания/прошлого периода? | просрочка ×2.3 к среднему | «в крупном сегменте крупные суммы» (тавтология) |
| **Адресность** | Понятно ли, КТО/ГДЕ именно? (имя, территория, менеджер) | «ГОСБ 38, топ-5 компаний = 60%» | «в среднем по таблице» |
| **Действенность** | Есть ли ручка? (вернуть, проверить, раскатить, перераспределить) | компании с высоким потенциалом среди ушедших | распределение системного кода загрузки |

Плюс два **жёстких стоп-правила** (нарушение = виджет не попадает в выдачу):

- **Стоп «тавтология»**: находка не должна следовать из определения данных (доля флага
  ~0%/~100% при null-семантике; лидерство самой крупной категории по абсолютной сумме
  без нормировки; мера против собственной производной).
- **Стоп «нечитаемо»**: разрез с кардинальностью, не влезающей в виз (>15 категорий без
  «прочие»), id без человеческой подписи, неопределённая единица измерения.

Это определение — контракт всей системы: скоринг (§6.2), критик (§6.4), eval (§10) и
фидбек (§9) измеряют одно и то же.

## 3. Целевая архитектура

### 3.1 Слои и модули

```
src/text2sql/

    model.py        #   pydantic-схемы SemanticTable/Measure/Dimension/JoinPath
    store.py        #   data_for_agent/semantics/<schema>.<table>.json, атомарная запись
    enrich.py       #   построение модели: интроспекция+статистика+ОДИН LLM-вызов+правки человека
  engine/           # СЛОЙ 2: data engine (pandas-first)
    request.py      #   AggregateRequest / QueryResultFrame (контракты)
    store.py        #   DataFrame Store: одна загрузка из БД, кэш в памяти, оптимизация dtype

    loader.py       #   БД-загрузчик: проекция колонок, where, гейт по памяти, dtype-схема

    widgets.py      #   WidgetSpec → фрагмент HTML/JS (Plotly) | таблица (Tabulator)
    page.py         #   сборка страницы: layout, офлайн-инлайн ассетов, TOC, фидбек-кнопки

  surfaces/         # ПОВЕРХНОСТИ
    ask.py          #   text2sql-агент (текущий graph/; финальный экспорт — как сейчас, из БД)

    investigate.py  #   /investigate — рекурсивная декомпозиция на engine/
```

### 3.2 Главный принцип срастания

**Все аналитические поверхности говорят с данными через один язык — `AggregateRequest`
(§5.1), который исполняется pandas-ом поверх DataFrame Store.** Дэш из 10 виджетов,
расследование на 4 уровня спуска и impact-анализ с сотней стратифицированных сравнений —
это **одна** загрузка из БД и дальше только память. Это и есть «не класть БД»: Greenplum
делает то, что умеет дёшево (отдать колонки с фильтром), pandas делает всё остальное.

Разделение труда с SQL-агентом: `/ask` остаётся про точный ответ и полный экспорт
(там исполнение в БД оправдано — пользователь просит именно выгрузку). Но резолв значений,
семантическая модель, join-знание и рендер — общие.

### 3.3 Персистентность: только JSON (решение владельца)

SQLite исключён: его файловые локи конфликтуют с жизненным циклом Jupyter-ядер (зависшее
ядро держит соединение → следующая сессия виснет на локе). Правила для всего персистентного:

- **Формат**: JSON (UTF-8, `ensure_ascii=False`, `indent=2`) — человекочитаемо, диффается
  в git, правится руками.
- **Гранулярность — файл на сущность**, а не один большой файл (меньше конфликтов,
  дешевле перезапись): `data_for_agent/semantics/<schema>.<table>.json`,
  `data_for_agent/feedback/<yyyy-mm>.jsonl` (append-only), `data_for_agent/golden/<name>.json`,
  `config_runtime.json` (как сейчас).
- **Атомарная запись обязательна**: `write tmp → os.replace(tmp, path)` — одна утилита
  `semantics/store.py:atomic_write_json()`, используется везде (заодно закрывает B-22 из
  HARNESS_REVIEW).
- **Без межпроцессных локов вообще**: единственный писатель — интерактивная сессия;
  append-only jsonl для фидбека не требует лока по построению.
- Кэш DataFrame'ов — **только память процесса** (см. §5.2), никакой дисковой сериализации
  в v1 (parquet-кэш — опция v2, тоже файлы без локов).

## 4. Слой 1: семантическая модель

### 4.1 Схема (pydantic; хранение — JSON-файл на таблицу)

```python
class Unit(str, Enum): money="money"; people="people"; count="count"; percent="percent"; days="days"; other="other"

class Measure(BaseModel):
    name: str                  # тех.имя колонки ИЛИ производной
    label: str                 # «Отток ФОТ»
    expr: str                  # pandas-выражение сборки: имя колонки | спец-тип производной
    derived: DerivedDef | None # rate/overdue/duration/impact — как в текущем build_derived
    agg: Literal["sum","mean","count","nunique"]
    unit: Unit
    good_direction: Literal["up","down","none"]   # рост — хорошо или плохо (окраска и формулировки!)
    priority: int = 0          # 0=главная мера таблицы … (LLM предлагает, человек правит)
    scope_mask: str | None     # "task_category == 'Предложение'" — pandas-query, где мера осмысленна

class Dimension(BaseModel):
    name: str; label: str
    card: int                  # кардинальность (из статистики)
    kind: Literal["category","geo","org","reason","status","channel","source","tool_flag"]
    id_partner: str | None     # gosb_id ↔ gosb_name
    top_values: list[str]      # до 25 значений для резолва фильтров

class Entity(BaseModel):
    name: str; label: str      # «Компания»
    key_col: str               # inn
    name_col: str | None       # company_name
    value_col: str | None      # колонка «ценности» (потенциал/ФОТ) для приоритизации

class TimeCol(BaseModel):
    name: str; label: str
    is_reporting: bool         # отчётная дата (ось анализа) vs событийная
    granularity: Literal["day","month"]
    fresh_until: date | None   # последний ПОЛНЫЙ период (детект неполного месяца!)

class JoinPath(BaseModel):
    to_table: str
    on: list[tuple[str,str]]
    cardinality: Literal["N:1","1:1"]           # только безопасные пути попадают сюда
    verified_at: date          # когда check_join подтвердил живой пробой
    enrich_cols: list[str]     # что полезно тащить: ["gosb_name","segment_name"]
    dedup: DedupDef | None     # свежая карточка на ключ (перенос текущей механики)

class SemanticTable(BaseModel):
    fqn: str; label: str; grain: str; row_meaning: str      # «одна строка = задача воронки»
    audience_hint: str         # «руководителям сети продаж» — контекст планировщика/нарратора
    time: list[TimeCol]; measures: list[Measure]; dimensions: list[Dimension]
    entities: list[Entity]; joins: list[JoinPath]
    outcome_measures: list[str]    # меры-ИСХОДЫ (закрытие, просрочка, срок) — для impact-режима (§8)
    tool_flags: list[str]          # булевы признаки «участия в инструменте» — кандидаты для /impact
    boring_cols: list[str]     # явный список «не анализировать» (тех.поля)
    question_bank: list[str]   # 5-10 вопросов, на которые таблица отвечает (§4.2)
    human_overrides: dict      # поля, правленные человеком: {path: value} — НЕ перетираются LLM
```

Файл: `data_for_agent/semantics/s_grnplm_..._uzp.uzp_dwh_sale_funnel_task.json`.
Загрузка всех моделей при старте — миллисекунды; правка руками — обычный редактор.

### 4.2 Как модель строится (enrich.py)

При `/refresh_metadata` для каждой таблицы:

1. **Детерминированная часть** (без LLM): интроспекция, статистика по адаптивному сэмплу
   (кардинальности, доли null/нулей, min/max дат, доля дробных), автоопределение
   `granularity`, `fresh_until`, пар `id_partner`, кандидатов `tool_flags` (булевы колонки
   с долей true в диапазоне 2–80%).
2. **Один структурный LLM-вызов** «семантическое профилирование» — вход: имена + описания +
   статистика + примеры; выход: labels, units, kinds, grain, row_meaning, приоритеты мер,
   производные (текущий `_BEHAV_SYS` вливается сюда), scope-правила, `good_direction`,
   `boring_cols`, `outcome_measures`, `question_bank` — «на какие 5–10 бизнес-вопросов
   отвечает таблица». Question bank — ключ к пользе: планировщик дэша (§5.3) обязан
   привязывать каждый виджет к вопросу из банка или из запроса пользователя.
3. **Правки человека** — интерактив в CLI (замена нынешнего `money_confirm`): показать
   модель, дать поправить единицы/приоритеты/скучные колонки. Правки пишутся в
   `human_overrides` и **никогда не перетираются** при следующем рефреше.
4. **Join-пути**: из join-графа берутся только пары, прошедшие `check_join` (N:1/1:1),
   с `verified_at`; для справочной стороны фиксируются `enrich_cols` и dedup-правило.

**Инкрементальность** (сценарий владельца «добавил колонку is_in_pipeline»): рефреш
сравнивает интроспекцию с моделью; новые колонки профилируются **отдельным дешёвым
LLM-вызовом только по ним**, модель дополняется, human_overrides сохраняются. Не нужно
пересобирать всё, чтобы система узнала о новом признаке.

### 4.3 Кросс-табличность

Семантическая модель хранит связи → engine умеет **обогащение в pandas**: любой
`AggregateRequest` может группироваться по колонке справочника
(`dim: "uzp_dim_gosb.gosb_name"`) — executor сам подгрузит справочник (маленький,
кэшируется навсегда в сессии), применит dedup и `merge` по верифицированному ключу.
Анти-fanout гарантирован конструктивно: в JoinPath попадают только N:1/1:1 пути, а dedup
применяется до merge. Это открывает дэшам и расследованиям стыковые инсайты (Д-3).

## 5. Слой 2: Data Engine (pandas-first) и дэшборды

### 5.1 Контракт `AggregateRequest`

Единственный способ для аналитических поверхностей получить цифры:

```python
class AggregateRequest(BaseModel):
    table: str
    measures: list[str]                 # имена мер из семантической модели
    dims: list[str] = []                # "segment_name" | "uzp_dim_gosb.gosb_name" | "time:month"
    filters: list[Filter] = []          # переиспользуем Filter из plan.model
    top_n: int | None = None            # + автоматическое «прочие» (остаток считается честно)
    order_by_measure: str | None = None
    compare: Compare | None = None      # декларативное сравнение периодов
    include_share: bool = True          # доля от итога

class Compare(BaseModel):
    time_col: str
    kind: Literal["prev_period","same_period_last_year","period_over_period_series"]
```

Executor (`executor.py`) — чистый pandas: `groupby(observed=True)` по подготовленным
categorical-колонкам, top_n + строка «прочие», доли, compare через маски периодов,
scope_mask меры применяется автоматически (правило «потенциал только для Предложений»
решено раз и навсегда на уровне движка). Результат — `QueryResultFrame`: маленький
DataFrame + метаданные (единицы, доли, compare-колонки) — на нём работают детекторы
и рендер.

### 5.2 DataFrame Store — «одна загрузка, много вычислений»

Правила (реализация `engine/store.py` + `loader.py`):

1. **Ключ кэша**: `(fqn, normalized_where, tuple(columns))`. Запрос с фильтром-подмножеством
   уже загруженного фрейма не ходит в БД — фильтруется в pandas от суперсета.
2. **Загрузка**: `SELECT <только нужные колонки> FROM t [WHERE ...]` через `run_export`
   (единственное обращение к БД за сценарий). Проекция колонок — из семантической модели
   (boring_cols и большие тексты не грузим — уже есть в `_load_columns`, переносится).
3. **Гейт по памяти**: перед загрузкой `estimate_row_count` × оценка ширины строки;
   бюджет — конфиг `PANDAS_MEM_BUDGET_GB` (дефолт 80 из 120). Превышение → предложить
   `--where` по свежему периоду или согласованный сэмпл (`WHERE random()<p` — тогда все
   виджеты получают плашку «по сэмплу N%»). С 120 ГБ это сработает редко, но инцидент
   «положили ноутбук» должен быть невозможен по построению.
4. **Оптимизация dtype при загрузке** (критично для 120 ГБ → «влазит много»):
   строковые разрезы → `category` (на низкокардинальных колонках это ×10–50 по памяти),
   целые → минимальный `Int*`, даты → `datetime64`, флаги → `boolean`. Схема dtype
   выводится из семантической модели, а не угадывается по данным.
5. **Инвалидация**: TTL на сессию + явный `/reload_data <table>`; справочники (reference,
   маленькие) кэшируются на всю сессию.
6. Дисковой персистентности кэша в v1 нет (см. §3.3); v2-опция — parquet-файлы в
   `workspace/cache/` (тоже без локов).

Следствие: **дэш из 10 виджетов, расследование на 4 уровня и impact-анализ на сотню
сравнений стоят БД ровно один SELECT.** Повторный дэш по той же таблице — ноль обращений.

### 5.3 `/dash` — от вопроса к дэшборду

Команда: `/dash <таблица|тема> <вопрос или задача> [--where ...]`.
**Вопрос обязателен** (Д-1). Если пользователь дал только таблицу — система не «майнит всё»,
а предлагает выбрать из `question_bank` (нумерованный список) или ввести свой. Режим
«полный обзор» остаётся, но как явный выбор `[0] Общий обзор`, и даже он планируется через
question bank, а не через «всё по всему».

Конвейер:

```
вопрос ─► ПЛАНИРОВЩИК (LLM №1, структурный) ─► DashboardSpec (черновик 8-14 виджетов)
      компактная семантическая модель          │
      + few-shots из золотого набора            ▼
                            ВАЛИДАЦИЯ + ОБОГАЩЕНИЕ (код): имена существуют, кардинальности
                            влазят, добавить обязательные виджеты (KPI-шапка, динамика
                            главной меры), выкинуть дубли
                                                    │
                                                    ▼
                            ИСПОЛНЕНИЕ (engine): 1 загрузка → N groupby в памяти (секунды)
                                                    │
                                                    ▼
                            INSIGHT ENGINE (§6): детекторы, скоринг, стоп-правила, ранжирование
                                                    │
                                                    ▼
                            КРИТИК (LLM №2): отбраковка тривиального, заголовки-выводы,
                            дозапрос «missing» (одна итерация)
                                                    │
                                                    ▼
                            НАРРАТОР (LLM №3): «Главное», выводы секций, «что делать»
                                                    │
                                                    ▼
                            RENDER: интерактивный HTML (Plotly+Tabulator) + MD-версия
```

Итого **3 LLM-вызова на дэш** (сейчас 5–6), все структурные. pandas-шаги — секунды.

### 5.4 `DashboardSpec` — декларативный контракт (продукт LLM-планировщика)

```python
class WidgetSpec(BaseModel):
    id: str
    question: str               # ← ОБЯЗАТЕЛЬНО: на какой вопрос отвечает виджет (Д-5)
    viz: Literal["kpi_row","bar","stacked_bar","line","area","pareto","heatmap",
                 "treemap","waterfall","scatter_quadrant","table","funnel"]
    request: AggregateRequest
    baseline: Literal["none","overall_avg","prev_period"] = "none"
    size: Literal["full","half"] = "half"

class DashboardSpec(BaseModel):
    title: str
    audience: str               # для тона нарратива
    question: str               # исходный вопрос (переформулированный)
    filters_global: list[Filter]
    widgets: list[WidgetSpec]   # после критика останется 5-9
```

Планировщику в промпт идёт **компактный рендер семантической модели** (labels, меры с
приоритетами и единицами, разрезы с кардинальностями, сущности, join-пути, question_bank,
fresh_until) + вопрос + 2–3 примера хороших DashboardSpec. Сырые данные ему **не** идут —
план строится по семантике, проверяется исполнением.

### 5.5 Каталог виджетов (render/widgets.py)

Каждый тип — детерминированная функция `(WidgetSpec, QueryResultFrame, InsightCard|None) → WidgetHTML`:

| viz | Когда | Реализация |
|---|---|---|
| `kpi_row` | шапка дэша | карточки: значение, Δ к пред.периоду (стрелка, окраска по `good_direction`!), спарклайн |
| `bar` | мера по разрезу ≤15 категорий | горизонтальный бар, пунктир базы, числа на барах, тултип с долей и n |
| `line`/`area` | динамика | месяцы, маркеры аномалий из детекторов, серая зона «неполный период» (fresh_until!) |
| `pareto` | концентрация | бары+кумулята, подсветка «сколько категорий = 80%» |
| `stacked_bar` | структура во времени | доли слагаемых по месяцам |
| `heatmap` | 2 разреза × мера | Plotly heatmap, аннотации, ≤8×8 + «прочие» |
| `treemap` | иерархия (разрез→сущность) | **нативный Plotly treemap** — интерактивный drill кликом (заменяет самодельный squarify) |
| `waterfall` | как складывается итог | нативный Plotly waterfall |
| `scatter_quadrant` | приоритизация (убыль×ценность) | подписи топ-точек, медианные линии |
| `table` | детальные списки («кто») | **Tabulator**: сортировка, поиск, фильтры по колонкам, закреплённая шапка, формат чисел/₽, цветовые бары в ячейках, экспорт CSV |
| `funnel` | воронка статусов | Plotly funnel |

Правила зашиты в рендер (не в LLM): деньги — с ₽ и разрядными пробелами; отрицательное
«хорошее» — зелёным по `good_direction`; >12 категорий — «прочие»; каждая ось подписана
label-ом из семантической модели.

### 5.6 Интерактивный HTML (инструменты: Plotly.js + Tabulator)

- **Plotly.js** — все графики: зум, пан, тултипы с точными значениями, вкл/выкл серий
  легендой, экспорт PNG; treemap/waterfall/funnel — нативные, drill-down в treemap кликом
  бесплатно.
- **Tabulator** (MIT) — все таблицы: клиентская сортировка/фильтрация/поиск, пагинация,
  группировка строк, форматтеры (деньги, проценты, прогресс-бары в ячейках), экспорт CSV.
- **Офлайн-режим обязателен** (банковский контур): `plotly.min.js` (~3.5 МБ) и
  `tabulator.min.js+css` кладутся в пакет (`render/assets/`) и **инлайнятся** в HTML.
  Файл 4–6 МБ — приемлемо; данные виджетов — inline JSON (`<script type="application/json">`),
  не base64-картинки.
- Каркас: шапка (вопрос, фильтры, период данных, плашка о неполном месяце), блок
  «🎯 Главное» с 3–5 пунктами-якорями на виджеты, CSS-grid сетка (full/half), у каждого
  виджета: **заголовок-вывод** (не «ТОП по X», а «60% потерь — три ГОСБа»), бейдж score,
  кнопки 👍/👎 (§9), сворачиваемый блок «как посчитано» (описание агрегата и фильтров —
  доверие через прозрачность).
- Лёгкий cross-filter (v2): клик по категории подсвечивает её в остальных виджетах
  (данные уже на клиенте, без пересчёта).
- MD-версия сохраняется параллельно (для почты/вики) со статичными PNG.

## 6. Слой 3: Insight Engine — гарантия «не фигни»

### 6.1 Детекторы работают на агрегатах

Все текущие детекторы (`rate_dev`, `money_conc`, `value_mismatch`, `interaction`,
`spikes`, `seasonality`, `repeat`) переписываются с сигнатуры «сырой df» на
`(QueryResultFrame, context)` — вход уже агрегирован executor-ом. Добавляются:

- `trend_break` — перелом тренда (последние 2–3 точки против экстраполяции предыдущих);
- `new_leader` — смена лидера разреза между периодами;
- `share_shift` — структурный сдвиг долей (сегмент вырос с 20%→35% в потерях);
- `benchmark_gap` — срез против медианы срезов той же меры («ГОСБ 38 хуже 80% ГОСБов»);
- `adoption_gap` — для tool_flags: разрыв использования между группами (питает /impact, §8);
- `data_quality` — константные колонки, вырожденные разрезы, неполный последний период →
  не в отчёт, а в **подавление** виджетов и плашки.

Каждый детектор возвращает `InsightCard`:

```python
class InsightCard(BaseModel):
    widget_id: str
    kind: str
    headline_facts: dict               # числа для заголовка/нарратива
    materiality: float                 # 0..1 — доля затронутых денег/объёма
    surprise: float                    # 0..1 — нормированное отклонение
    addressability: float              # 0..1 — есть имя/территория/сущность?
    actionability: float               # 0..1 — по типу детектора
    score: float
    stop_flags: list[str]              # «тавтология», «нечитаемо», «дубль» → не показывать
```

### 6.2 Формула полезности (единая, конфигурируемая)

```
score = (w_m·materiality + w_s·surprise + w_a·addressability + w_x·actionability)
        · money_boost      # ×1.5 если мера unit=money
        · focus_boost      # ×1.3 если виджет напрямую отвечает на вопрос пользователя
        · penalty_dup      # ×0.5 за каждый повтор того же значения среза в уже отобранных
```

Веса и пороги — в **одном** конфиге `insight/scoring.yaml` с тремя профилями:
`strict` (только громкое, для руководителя), `default`, `explore` (для аналитика).
Пороги материальности — **адаптивные**: `min_rows = max(30, 0.5% строк)`,
`min_money_share = 3%` (не абсолютные числа). Это закрывает Д-4.

### 6.3 Стоп-правила (детерминированные, до критика)

- тавтология: доля ≥97%/≤3% у rate-меры без денежного веса; лидер по сумме = лидер по
  количеству при сопоставимых средних; мера против собственной производной;
- нечитаемость: нет label/unit; кардинальность не влазит; «прочие» >70%;
- дубль: та же пара (мера, значение среза) уже показана виджетом с большим score;
- неполный период: «падение в последнем месяце» при незакрытом месяце (fresh_until) →
  подавить, показать плашку.

### 6.4 LLM-критик (вызов №2)

Вход: карточки (headline_facts + question виджета + score), вопрос пользователя, аудитория.
Выход (структурный):

```json
{"verdicts": [{"id":"w3","keep":true,"headline":"Три ГОСБа дают 61% потерь ФОТ",
               "reason_if_drop":null}, ...],
 "ordering": ["w1","w3","w7"],
 "missing": ["нет разбивки по времени — когда началось?"]}
```

Правила критику: держать 5–9 виджетов; отбраковывать тавтологии, проскочившие правила;
у каждого выжившего — заголовок-вывод с числом. Поле `missing`: если дыра выражается одним
`AggregateRequest` — движок дозапрашивает (в памяти это бесплатно) и виджет добавляется.
Одна итерация, не цикл.

### 6.5 Нарратор (вызов №3)

Как текущий `narrate`, но: (а) только выжившие после критика; (б) получает **связи** —
предвычисленный список «значение X встречается в виджетах w1,w3,w5» (сквозной сюжет:
«ГОСБ 38 фигурирует и в потерях, и в просрочке — похоже, системная проблема территории»);
(в) в «Главном» — 3–5 пунктов вида «факт с числом → значит → что сделать»;
(г) запрет матжаргона — как сейчас.

## 7. Расследования 2.0

Расследование = **направленный обход того же ядра**: дэш — ширина, расследование — глубина.

### 7.1 Конвейер

```
вопрос ─► FRAME (LLM №1): target-мера, направление (loss/gain/top), кандидаты-разрезы,
          кандидаты-причины, сущность, мера ценности — всё из семантической модели
              │
              ▼
   РЕКУРСИВНЫЙ СПУСК (pandas, без LLM): на каждом уровне
     • для всех разрезов-кандидатов: декомпозиция вклада (top_n=12)
     • выбираем разрез с максимальной концентрацией (top_share)
     • спускаемся в топ-1..2 узла (маска), убираем использованный разрез
     • стоп: глубина 4 / вклад узла <15% / < min_rows → честное «дальше размазано»
     СПУСК МОЖЕТ ПЕРЕСЕКАТЬ ТАБЛИЦЫ по join-путям модели (merge в pandas):
     сегмент из ЕПК, названия ГОСБ из справочника — стыковые расследования (Д-3)
              │
              ▼
   ВРЕМЯ (pandas): динамика target по месяцам в найденном «горячем» пути;
   trend_break → «когда началось»; сравнение до/после точки перелома
              │
              ▼
   ПОЧЕМУ (pandas): разрезы-причины С НОРМИРОВКОЙ — lift против базового распределения:
   «в горячем срезе причина Х встречается ×2.4 чаще, чем в среднем», а не
   «преобладает Иное (80%)», которое преобладает везде
              │
              ▼
   КТО (pandas): сущности внутри горячего пути + ценность + Парето + квадрант приоритета
              │
              ▼
   ГИПОТЕЗЫ-ПРОВЕРКИ (LLM №2, опц.): LLM предлагает ≤3 проверяемых гипотезы в виде
   AggregateRequest («если сезонность — тот же месяц прошлого года покажет то же»);
   движок исполняет В ПАМЯТИ (бесплатно); результат = подтверждено/опровергнуто
              │
              ▼
   СИНТЕЗ (LLM №3): ответ, причинная цепочка с числами, действия, явный блок
   «что НЕ подтвердилось» (опровергнутые гипотезы — тоже ценность и честность)
```

### 7.2 Выход

Тот же render-движок: дерево спуска — **Plotly treemap/icicle с drill-down по клику**
(иерархия уже посчитана), временной график с маркером перелома, таблица «кто» — Tabulator
с сортировкой по ценности, waterfall итога. Структура: Ответ → Когда началось → Где
сосредоточено → Почему (lift) → Кто (+приоритет) → Проверенные гипотезы → Что делать.

### 7.3 Смычка поверхностей

1. `/ask` → результат → **«дэш по этому»**: одобренный план агента = готовые table+filters
   для `DashboardSpec.filters_global`.
2. Виджет дэша → «расследовать» → `/investigate` стартует с target и фильтрами виджета.
3. Расследование само генерирует `--where` из вопроса («к 2026» → фильтр по времени)
   тем же резолвом фильтров, что у агента.
4. Вопрос про эффективность/влияние/пилот → роутинг в `/impact` (§8).

## 8. Режим «Эффективность инструмента» (/impact) — новый обязательный сценарий

### 8.1 Постановка (сценарий владельца, дословно)

> В таблицу `sale_funnel` добавлен признак `is_in_pipeline` (задача участвует в инструменте).
> Промт: «разбери признак участия задачи в пайплайне. Как я могу доказать эффективность
> инструмента». Ожидание: система посчитает, где используется и где нет; найдёт территорию
> или группу сотрудников, которые используют; предложит её как пилотную группу; покажет,
> что задачи там закрываются лучше; и сделает вывод: «инструмент реально эффективнее,
> при экстраполяции на все задачи % закрытия вырастет до X».

Это отдельный класс анализа — **сравнение исходов с инструментом и без (treatment vs
control)** — со своей методологией и своими ловушками. Главная ловушка, которую система
обязана обходить и честно называть: **самоотбор** («те, кто взял инструмент, возможно,
и без него работали лучше» — наивное сравнение это не отличит).

### 8.2 Конвейер `/impact <таблица> <вопрос>` (модуль `insight/impact.py` + `surfaces/impact.py`)

```
вопрос ─► FRAME (LLM №1): по семантической модели определить
          • treatment: булев признак (из tool_flags; в примере — is_in_pipeline)
          • outcomes: меры-исходы (из outcome_measures: доля закрытия, срок, просрочка;
            + денежный исход, если есть)
          • unit: что считаем «объектом» (задача) и КТО решает об использовании
            (группирующие разрезы: сотрудник, ГОСБ/ТБ, сегмент, тип задачи)
          • time_col: для динамики внедрения
              │
              ▼
   ШАГ 1 — ADOPTION (pandas): картина использования
     • общая доля задач с инструментом; динамика по месяцам (кривая внедрения)
     • adoption по каждому группирующему разрезу (территория/сотрудник/тип)
     • детект НАТУРАЛЬНЫХ ГРУПП: «пилоты» = группы с adoption ≥ 60% и n ≥ min_rows;
       «контроль» = группы с adoption ≤ 10%, похожие на пилотов по профилю
       (размер, микс сегментов/типов задач, исход в ДО-периоде)
              │
              ▼
   ШАГ 2 — НАИВНОЕ СРАВНЕНИЕ (pandas, помечается как наивное): исходы задач
     с флагом vs без по всей таблице. Показываем, но подписываем: «без поправки
     на состав — может завышать/занижать эффект»
              │
              ▼
   ШАГ 3 — ЧЕСТНОЕ СРАВНЕНИЕ (pandas), три независимых взгляда:
     a) СТРАТИФИКАЦИЯ: сравнение с/без ВНУТРИ однородных ячеек
        (территория × тип задачи × месяц), эффект = взвешенное среднее
        по-страта-разниц (веса = размер страты). Убирает смещение состава.
     b) ПИЛОТ vs КОНТРОЛЬ: исходы пилотных групп против контрольных
        (сопоставленных по профилю из шага 1) — сравнение на уровне ГРУПП,
        устойчивое к индивидуальному самоотбору внутри группы
     c) ДО/ПОСЛЕ (diff-in-diff), если у внедрения есть дата старта:
        Δисход(пилоты, после−до) − Δисход(контроль, после−до).
        Самый сильный аргумент: «у пилотов стало лучше ИМЕННО после внедрения,
        у контроля за то же время — нет»
              │
              ▼
   ШАГ 4 — ПРОВЕРКИ ЧЕСТНОСТИ (pandas, обязательные):
     • self-selection check: исходы будущих пилотов в ДО-периоде vs контроль
       (если пилоты и раньше были лучше — сказать это прямо и опереться на (c), а не (a))
     • устойчивость: в какой доле страт эффект в ту же сторону (норма ≥ 70%)
     • достаточность: min объём с/без в каждой опоре; иначе вердикт «данных мало»
     • placebo-исход: мера, на которую инструмент влиять не должен — эффект там ≈ 0
              │
              ▼
   ШАГ 5 — ЭКСТРАПОЛЯЦИЯ (pandas): консервативная и честная
     • эффект = интервал [страт-оценка … DiD-оценка], НЕ точка
     • прирост = uplift_пп × (1 − adoption) × объём задач
     • формулировка: «закрытие 62% → ~69–72% (+7…10 п.п.), это ≈ +N закрытых задач
       в месяц» + явный список допущений (эффект переносится на неохваченные группы;
       без насыщения; состав задач не меняется)
              │
              ▼
   СИНТЕЗ (LLM №2): вердикт строго одной из трёх форм —
     «эффект подтверждается» (какими из трёх взглядов, каким числом),
     «эффект не подтверждается / отрицательный»,
     «данных недостаточно — вот что собрать» (какие группы/период докопить).
   + рекомендация пилота (конкретная группа с числами) + план раскатки + риски
```

LLM-вызовов: **2** (frame + синтез). Вся статистика — pandas на одном загруженном фрейме.

### 8.3 Выход `/impact` (HTML, тот же рендер)

1. **Вердикт-карточка** (зелёная/красная/серая): «Инструмент работает: +8 п.п. к закрытию
   (диапазон 7–10 п.п. по трём методам)».
2. **Кривая внедрения** (line) + **карта adoption** (bar по территориям: кто уже в пилоте).
3. **Сравнение исходов**: наивное и честное рядом (grouped bar с подписью методов) —
  прозрачность вместо магии.
4. **Пилот vs контроль**: таблица (Tabulator) сопоставленных групп с исходами до/после;
   line-график DiD (две линии, маркер даты внедрения).
5. **Разброс эффекта по стратам** (bar со знаком: где инструмент работает лучше/хуже —
   само по себе инсайт «где раскатывать в первую очередь»).
6. **Waterfall экстраполяции**: текущее закрытие → +эффект на неохваченных → целевое.
7. **Блок честности**: результаты проверок шага 4 простым языком («пилоты и до внедрения
   были лучше на 1.5 п.п. — эффект скорректирован», «в 82% сочетаний территория×тип
   эффект в плюс»).

## 10. Eval полезности

1. **Структурные проверки** (без LLM): каждый виджет привязан к question; нет стоп-флагов
   в выдаче; заголовки содержат числа; единицы у всех мер; ≤9 виджетов; **KPI-числа дэша
   сходятся с независимым pandas-пересчётом** (числовая корректность).
2. **LLM-судья по рубрике §2** (для регрессионного A/B сравнения версий): оценка каждой
   секции по 4 осям 0–2, агрегат по дэшу; прогон на золотом наборе → тренд.
3. **Анти-фигня-кейсы**: специально скучные таблицы (нулевые меры, один сегмент,
   константы) — система обязана выдать короткий честный дэш «показывать почти нечего,
   вот 2 факта», а не стену шума.
4. Кейсы расследований с **подложенной причиной** (сидер уже умеет) — recall обязан быть 3/3.
5. **Impact-кейсы с подложенным эффектом**: сидер генерит is_in_pipeline с известным
   uplift (+8 п.п. у части групп) И отдельный кейс с чистым самоотбором (флаг у заведомо
   сильных сотрудников, эффекта нет) — система обязана первый подтвердить с точностью
   ±2 п.п., второй — отклонить с формулировкой про самоотбор. Это главный тест честности.

## 11. План реализации (для implement-агента)

### Фаза 1 — фундамент
1. `semantics/`: pydantic-схемы, **JSON-store** (файл на таблицу, `atomic_write_json`),
   enrich с одним LLM-вызовом, инкрементальное дополнение при новых колонках, конвертер
   из текущих CSV, интерактивная правка человека (human_overrides).
2. `engine/`: AggregateRequest, **pandas-executor** (groupby/top_n+прочие/доли/compare/
   scope_mask/join-обогащение), DataFrame Store (кэш по ключу, dtype-оптимизация,
   гейт по памяти `PANDAS_MEM_BUDGET_GB`).
3. `render/`: vendored Plotly+Tabulator, каталог виджетов, сборка страницы, офлайн-инлайн.
   Приёмка: текущий отчёт собирается на новом рендере.

### Фаза 2 — /dash
4. DashboardSpec + планировщик (LLM №1) + валидация/обогащение.
5. Insight engine: детекторы на агрегатах, scoring.yaml с профилями, стоп-правила.
6. Критик + нарратор + композиция. `/build_business_report` → алиас `/dash <t> "общий обзор"`.
   Приёмка: eval §10.1+10.3 зелёный; на 6 демо-таблицах ≤9 виджетов, каждый с вопросом
   и заголовком-выводом; KPI сходятся с пересчётом.

### Фаза 3 — /investigate 2.0 и /impact
7. Рекурсивный спуск на engine (в т.ч. кросс-табличный merge), время+trend_break, lift-почему,
   гипотезы-проверки, treemap с drill.
8. **`insight/impact.py`**: adoption → страты → пилот/контроль → DiD → проверки честности →
   экстраполяция; `surfaces/impact.py` + роутинг из вопросов; сидер impact-кейсов.
   Приёмка: §10.4 (3/3) и §10.5 (подложенный эффект найден ±2 п.п., самоотбор отклонён).

### Фаза 4 — смычка и фидбек
9. «Дэш по результату /ask», «расследовать виджет», /feedback + золотой набор +
   few-shots-подтяжка, LLM-судья в eval.

### Бюджет LLM-вызовов (gigachat-3-ultra, структурный вывод)

| Сценарий | Вызовы | pandas-время | обращений к БД |
|---|---|---|---|
| refresh таблицы | 1 (+1 на новые колонки) | — | интроспекция+сэмпл |
| /dash | 3 (план, критик, нарратор) | секунды | 1 SELECT |

### Определение успеха (продуктовое)
Бизнес-пользователь на незнакомой системе таблице задаёт свой вопрос через `/dash` и
получает за ~1 минуту страницу, где каждый блок отвечает на понятный вопрос, главные
выводы названы с числами и именами, есть минимум одно действие — и ни одного блока «зачем
мне это?». А на вопрос «докажи, что инструмент работает» система отвечает вердиктом,
который не стыдно показать руководству: с методами, проверками честности и интервалом
эффекта, а не одной красивой цифрой.

## 13. Приложение: находки аудита текущих аналитических функций

Перенесено из `HARNESS_REVIEW.md` (аудит кода от 2026-07-07) с **сохранением сквозной
нумерации B-***. Каждая находка привязана к разделу этого документа, где она решается —
это чек-лист для приёмки реализации: находка считается закрытой, только если указанный
механизм реально её устраняет.

Из положительного (тоже перенос из аудита): отчёт и расследование уже построены по
правильному принципу — **числа считает pandas детерминированно, LLM решает только
смысловые задачи** (роли колонок, подписи, нарратив). Для слабой модели это единственно
верное распределение труда; новая архитектура этот принцип сохраняет и усиливает
(pandas становится общим движком §5, LLM-вызовы сокращаются и становятся структурными §8).

### 13.1 Конкретные баги

**B-3 [критичный]. `/build_business_report` и `/investigate` без `--where` грузят ВСЮ
таблицу в pandas.** `report/builder.py:33-39` — `run_export(sql, enforce_limit=False)`:
ни оценки размера (`estimate_row_count` есть в адаптере и не используется), ни
подтверждения, ни автосэмплирования. На демо-контейнере 4–6 тыс. строк — норм; на проде —
предсказуемый инцидент.
→ **Решение: §5.2 DataFrame Store.** Сама загрузка в pandas — осознанное решение владельца
(120 ГБ RAM, БД не грузим вычислениями), но «слепой» она быть перестаёт: гейт по памяти
`PANDAS_MEM_BUDGET_GB` через `estimate_row_count` × ширину строки, проекция только нужных
колонок из семантической модели, dtype-оптимизация (category даёт ×10–50), кэш фреймов
(повторный дэш = ноль обращений к БД), согласованный сэмпл с плашкой при превышении бюджета.

**B-15 [сопровождаемость], аналитическая часть мёртвого кода (~700 строк):**
- `report/plan.py`: `select_primary_metrics`, `candidate_specs`, `select_specs`,
  `_default_pick`, `_spec_label`, `run_spec` — весь старый спек-конвейер не вызывается
  билдером;
- `report/core.py`: `top_n`, `top_n_count`, `trend`, `concentration`, `period_compare`,
  `flag_breakdown`, `kpi`, `_md_table` — вызываются только из мёртвого `run_spec`;
- `report/investigate.py:231-258`: функция спуска `_drill` + датакласс `Level` — не
  вызываются оркестратором;
- `report/patterns.py:116`: `new_and_gone` — отключён, но оставлен;
- `report/builder.py:159`: `angle = ""` — рудимент удалённого `select_specs`.
→ **Решение: карта миграции §12** — модули замещаются целиком, мёртвое не переносится.
Отдельно: идея `_drill` (многоуровневый спуск) **возвращается в строй** как рекурсивный
спуск §7.1 — она была правильной, просто не доведена до вызова.

**B-16 [косметика]. Вывод команд отчёта в CLI:** `cli.py:185,233` —
`print(f"...(секций: {a}, графиков: {b}, строк: {c:,})".replace(",", " "))` заменяет
**все** запятые, включая разделители перечисления: получается «секций: 5  графиков: 3».
→ **Решение:** тривиальный фикс при переносе команд на новые поверхности (`surfaces/dash.py`
и др.) — форматировать число до подстановки (`f"{c:,}".replace(",", " ")`), а не всю строку.

**B-17 [косметика]. Мёртвое условие в waterfall расследования:** `investigate.py:475` —
внутренний тернарник `if len(items) > 9` находится под внешним `if len(items) > 12`,
всегда истинен; приоритет операций делает выражение трудночитаемым.
→ **Решение:** самодельный waterfall уходит целиком — §5.5, нативный Plotly waterfall.

**B-25 [сопровождаемость]. Мутация данных и глобального состояния:**
`metrics.build_derived` и `mining.mine` пишут служебные колонки (`__rate_i`, `__row`)
прямо в переданный df; `report/core.py:14-21` при **импорте** меняет глобальный matplotlib
(`matplotlib.use("Agg")`, `sns.set_theme`, `plt.rcParams`) — вид графиков всего ноутбука
пользователя.
→ **Решение:** производные колонки строятся движком при загрузке фрейма в DataFrame Store
(§5.2, по определениям `Measure.derived` из семантической модели §4.1) — executor
агрегатов ничего не мутирует; matplotlib уходит из интерактивного пути вообще (§5.5–5.6,
Plotly), остаётся только в генерации статичных PNG для MD-версии — с локальным rc-контекстом,
не глобальным.

### 13.2 Архитектурные слабости бизнес-отчёта (ex-§4.6 аудита)

1. **Пять последовательных LLM-вызовов** (labels → roles → behaviors → [focus] → narrate)
   с пересекающимися входами. Под 5-секундный rate-limit GigaChat это ≥25–45 с чистого
   ожидания; плюс рассогласования («профайлер сказал metric, labels сказал other»).
   → **Решение: §4.2** — один enrich-вызов на таблицу при рефреше (кэш в JSON навсегда) +
   **§5.3** — три вызова на дэш (план, критик, нарратор). Бюджет — таблица в §11.
2. **Результат профилирования не кэшируется** — повторный отчёт по той же таблице заново
   спрашивает LLM обо всём.
   → **Решение: §4** — профиль таблицы = семантическая модель, персистентный JSON-артефакт;
   рантайм отчёта его только читает.
3. **Пороговые константы рассыпаны по коду** (0.05/1.3/1.43/0.25/1.8/0.08/4×…) без единого
   конфига чувствительности; на другой таблице дают либо пустой отчёт, либо стену мусора.
   → **Решение: §6.2–6.3** — единый `insight/scoring.yaml` с профилями
   strict/default/explore и адаптивными порогами от объёма данных.
4. **Секции отчёта не имеют сквозной «важности»** — порядок задан типами, а не score;
   главная находка может оказаться в середине.
   → **Решение: §6.2 + §6.4** — единый score ранжирует все виджеты, критик задаёт
   финальный `ordering`.
5. **Нарратор не видит связей между секциями** (один и тот же ГОСБ в трёх находках) —
   сквозной сюжет собрать не из чего.
   → **Решение: §6.5** — нарратору передаётся предвычисленный список пересечений значений
   срезов между виджетами.

### 13.3 Архитектурные слабости расследований (ex-§4.7 аудита)

Концептуально это лучший модуль текущего кода (frame → decompose → tree → why → who →
synth — честная методология аналитика). Слабости:

1. **Один уровень «где»**: несмотря на наличие (мёртвой) функции спуска `_drill`, реальное
   дерево — только `primary_dim → entity`. «Сегмент → территория → холдинг → компания»
   не строится, хотя `_decompose` это умеет.
   → **Решение: §7.1** — рекурсивный спуск до глубины 4 с порогами материальности и честным
   стопом «дальше размазано»; плюс спуск может пересекать таблицы по join-путям модели.
2. **«Почему» = преобладающее значение разреза-причины, без сравнения с базой**: если 80%
   всех строк имеют причину «Иное», то «преобладает Иное (80%)» — не инсайт.
   → **Решение: §7.1, шаг ПОЧЕМУ** — lift-нормировка против базового распределения
   («в горячем срезе причина Х встречается ×2.4 чаще, чем в среднем»); механика уже
   отлажена в `_rate_deviations` майнинга — переиспользуется.
3. **Нет времени**: «где потеряли к 2026» напрашивается на «когда именно» — динамика потерь
   с детекцией перелома отсутствует (в mining есть `spikes`, в investigate — нет).
   → **Решение: §7.1, шаг ВРЕМЯ** — обязательный этап: динамика target в горячем пути,
   `trend_break`, сравнение до/после точки перелома.
4. **Дублирование `_prepare` с билдером** — те же 3 LLM-вызова (labels/roles/behaviors)
   выполняются заново при каждом расследовании.
   → **Решение: §4** — оба сценария читают одну семантическую модель; `_prepare`
   исчезает как класс.
5. **`_frame` fallback `num_cols[0]`** — при промахе LLM целевой величиной расследования
   становится произвольная первая числовая колонка.
   → **Решение: §7.1 FRAME** — target выбирается только из `measures`/`outcome_measures`
   семантической модели; при невозможности сматчить — уточняющий вопрос пользователю,
   а не молчаливый произвол.

### 13.4 Eval аналитических функций (ex-§4.8 аудита, фрагмент)

Текущий `eval/test_report.py` проверяет отчёт только структурно («есть 4 секции», «есть
графики», «нет матжаргона») — но не по содержанию: правильные ли числа в KPI, полезны ли
находки, не мусор ли секции.
→ **Решение: §10** — числовая сверка KPI с независимым pandas-пересчётом, LLM-судья по
рубрике полезности §2, анти-фигня-кейсы на скучных таблицах, расследования с подложенной
причиной (recall 3/3) и impact-кейсы с подложенным эффектом и подложенным самоотбором.
