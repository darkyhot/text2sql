# Text2SQL Agent — Design

> LLM-first агент text2sql в стиле hermes/openclaw: тонкий tool-using цикл на
> LangGraph, минимум предвычисленных артефактов, всё что можно — добывается
> вживую инструментами. Никакой вшитой бизнес-логики и детерминизма в принятии
> решений; код даёт только **инструменты** и **предохранители**. Харнес обязан
> работать на любых таблицах любой БД.

Статус: draft v1. Код по этому документу ещё не написан.

---

## 1. Цель и контекст

Пользователь задаёт вопрос на естественном языке → агент возвращает **план**
(на естественном языке). Пользователь окает или корректирует. По «ОК» агент
синтезирует SQL, валидирует, выполняет и выгружает результат в
`workspace/last_query.csv`. Цикл коррекции повторяется до согласия.

**Среда:**
- Сейчас: LLM = DeepSeek `deepseek-v4-flash`; БД = Postgres 15 (контейнер
  `db-agent-test-pg`). Схема `s_grnplm_*` указывает, что **прод-СУБД — Greenplum**.
- Прод: LLM = `gigachat-3-ultra`. → И LLM, и SQL-диалект скрыты за адаптерами.

**Жёсткие ограничения (от заказчика):**
- Полностью LLM-first (как hermes/openclaw). Решает модель, не код.
- Модели слабые → **короткий контекст, больше мелких шагов в графе**.
- Никакого детерминизма/хардкода под конкретные таблицы. Generic на любую БД.
- Агент может делать доп. запросы к БД (read-only) и пользоваться метаданными.
- **Агент не должен размножать строки.** PK физических нет — только составные,
  выводимые из данных.

## 2. Доменная модель (текущий датасет — для тестов, НЕ хардкодить)

Схема `s_grnplm_ld_salesntwrk_pcap_sn_uzp`, 4 таблицы:

| Таблица | grain | роль | PK (эвристика по сэмплу) |
|---|---|---|---|
| `uzp_data_epk_consolidation` | client | fact/dim | `epk_id` |
| `uzp_dim_gosb` | organization | reference | составной `(tb_id, old_gosb_id)` |
| `uzp_dwh_fact_outflow` | event | event | составной `(gosb_id, inn, report_dt)` |
| `uzp_dwh_sale_funnel_task` | task | fact | `task_code` |

Эти факты — **входные данные тестов**, а не конфиг агента. Агент должен выводить
их инструментами на любой схеме.

## 3. Эталонные сценарии (acceptance + первый eval-набор)

1. **«Сколько всего ТБ и ГОСБ?»**
   `select count(distinct tb_id), count(distinct new_gosb_id) from uzp_dim_gosb`
   — урок: семантически верная колонка (`new_gosb_id` = актуальный id), **не** PK
   `old_gosb_id`. Чистое семантическое рассуждение + при сомнении проба.

2. **«Сумма оттоков по дате и названию ГОСБ»**
   `... fact_outflow JOIN dim_gosb ON fact.tb_id=dim.tb_id AND fact.gosb_id=dim.old_gosb_id`
   — урок: **join по полному составному PK измерения**; `fact.gosb_id → dim.old_gosb_id`
   (не `new_gosb_id`). По одному `gosb_id` — размножение строк. Главная ловушка.

3. **«Сколько задач по фактическому оттоку в феврале 2026?»** — **две** валидные
   трактовки (`fact_outflow.is_task` vs `sale_funnel_task.task_subtype ILIKE
   '%фактический отток%'`). Урок: неоднозначность **показывать**, не угадывать.

Из них — три инварианта качества: **(A)** верная колонка, **(B)** join без
размножения строк, **(C)** явная неоднозначность.

## 4. Архитектурный принцип

Агент = LLM-цикл «рассуждение → вызов инструмента → наблюдение», обёрнутый в
LangGraph ради персистентного состояния, прерываний (human-in-the-loop) и
ретраев. Код **не** содержит правил выбора таблиц/колонок/джойнов — он содержит
инструменты и предохранители. Метаданные — это **контекст**, который подаётся
лениво и порционно, а не движок принятия решений.

Принцип контекста: **Tier-0 всегда, Tier-1 после выбора таблиц, Tier-2 по
явному запросу узла.** Это и есть «короткий контекст, больше шагов».

---

## 5. Инструменты агента (узкие, generic, за диалект-адаптером)

| Инструмент | Вход → выход | Заметки |
|---|---|---|
| `search_tables(query)` | NL → топ-K таблиц (имя, описание, grain, роль) | **retrieval-first** (BM25/вектор). Масштабируется на тысячи таблиц |
| `describe_table(t)` | таблица → колонки: dtype, класс, PK?, описание, sample_values | Tier-1. Кэш на сессию |
| `distinct_values(col, like?)` | колонка → топ-N значений (вживую) | Резолв enum/label. Заменяет статический value-резолв |
| `probe_sql(select)` | read-only SELECT/EXPLAIN → строки/план | Авто-LIMIT, timeout, cost-потолок, только SELECT |
| `check_fanout(plan)` | структура join → вердикт N:1 / fan-out + объяснение | Ядро защиты от размножения строк (см. §8) |

`rule_registry.json` и `semantic_lexicon.json` как статические файлы **из
критического пути убираются**. Их роль выполняют `distinct_values`/`probe_sql`
вживую. Подсказки — динамические (§8), не реестр.

## 6. Предвычисленный слой (минимальный; только дорогое/нестабильное вживую)

Оффлайн-джоб (`catalog/refresh.py`, generic на любую БД) кэширует **только**:

1. **Каталог** — таблицы (имя/описание/grain/роль) + BM25-индекс для `search_tables`.
2. **Колонки** — dtype, semantic_class (generic-эвристика по типу+имени+
   кардинальности), описание, до 12 sample_values из реальных данных.
3. **Составные PK** — как **гипотезы**; агент подтверждает их живой пробой
   (`check_join`). См. ниже про сидирование.
4. **Граф join-кандидатов** (`join_candidates.json`) — рёбра по совместимости
   типов + **пересечению значений** (живая проба overlap). Именно overlap
   разрешает ловушку old/new: для fact↔dim видно `old_gosb_id↔gosb_id` overlap 51
   против `new_gosb_id↔gosb_id` overlap 4 — LLM читает evidence, а не гадает.

НЕ генерируем: value-профили, лексикон, rule_registry. Это «минимум артефактов».

**Сидирование при синтетических данных (важный урок реализации).** Тестовый
контейнер содержит сгенерированные данные: числа/даты/id/булевы — реальные, а
строковые колонки **рандомизированы** (вместо «ММБ/КСБ» — мусор). Два следствия:
- На синтетике перебор PK по сэмплу **несостоятелен**: случайная колонка
  (`author_login`) уникальна на 200 строках и побеждает настоящий составной ключ
  `(tb_id, old_gosb_id)`. Поэтому PK (и описания/grain) **сидируются из
  кураторской метадаты**, а сэмпл-эвристика — только fallback для новых таблиц.
  На реальном Greenplum эвристика работает; сидирование — страховка.
- Строковые фильтры из метадаты в контейнере **не находятся** (значения
  рандомизированы). Для eval это значит: проверяемы запросы на id/числах/датах/
  флагах; строковые value-фильтры верифицируемы только на проде/реальных данных.

## 7. Граф (LangGraph) и состояние

### 7.1 Узлы

```
[scope]            search_tables → таблицы-кандидаты; детект неоднозначности
   │
[clarify?] ──ask──► interrupt: спросить юзера (≥2 валидных трактовок)
   │
[explore]          describe_table / distinct_values / probe_sql — собрать факты
   │
[grain&agg]        явное рассуждение о грануле, метриках, distinct/NULL/границах дат
   │
[join_advisor]     мап FK→PK измерения + check_fanout; динамическая подсказка
   │
[plan]             СТРУКТУРИРОВАННЫЙ план → рендер на NL → interrupt (показать)
   │  ◄── correction: точечный re-run затронутого узла (persist state)
   │ ОК
[synth]            структурированный план → SQL (диалект-адаптер)
   │
[validate]         план↔SQL консистентность + EXPLAIN + повторный check_fanout
   │
[execute]          → workspace/last_query.csv
```

### 7.2 Состояние (персистентное между ходами)

```
question, history[]
candidate_tables[], chosen_tables[]
facts{}            # результаты explore (кэш describe/probe)
plan: StructuredPlan   # см. §9 — контракт, NL это рендер
sql, validation{}, fanout_verdict
step_budget, probe_budget   # governors (§11)
trace[]            # полный лог узлов/инструментов (§10)
```

Коррекция плана меняет только нужную часть состояния и перезапускает один узел —
без переэксплоринга. Дёшево для слабой модели.

## 8. Защита от размножения строк (центральный инвариант B)

**Принцип:** *при join к reference/dim-таблице условие обязано покрывать её
полный составной PK; FK факта мапятся на PK-колонки измерения; кардинальность
N:1 подтверждается пробой.*

`join_advisor`:
1. берёт PK dim из кэша (**как гипотезу**);
2. мапит FK факта → PK-колонки dim по имени/семантике/пересечению значений
   (`distinct_values`/`probe_sql`) — это же разрешает ловушку `old/new`:
   правильный мап тот, что уникален И пересекается значениями с фактом;
3. `check_fanout` — дешёвый путь: проверка по кэшу (PK/unique_perc); если
   сомнительно — живая проба **на сэмпле**, не full scan:
   `select <keys>, count(*) ... group by <keys> having count(*)>1 limit 1`;
4. формирует **динамическую подсказку**: «join по (tb_id, old_gosb_id) — полный
   ключ; по одному gosb_id будет размножение».

Повторно `check_fanout` гоняется в `[validate]` на финальном SQL. Fail →
агент доращивает ключ или агрегирует.

> Важно: fan-out-гард ловит только структурный дубль. Семантические ошибки
> агрегации (SUM по дублям, COUNT vs COUNT DISTINCT, NULL, границы дат) —
> ответственность узла `[grain&agg]`, не fan-out.

---

## 9. Контракт плана (структура, а не текст)

NL-план, который видит пользователь, — **рендер** структурированного объекта.
Согласие пользователя привязано к структуре, и `synth` обязан ей соответствовать
(проверяется в `[validate]`). Это закрывает риск «план окнули, а SQL уехал».

```
StructuredPlan:
  tables:   [{ref, alias}]
  joins:    [{left, right, on:[(lcol,rcol)], kind, fanout_safe:bool}]
  filters:  [{col, op, value, resolved_via}]   # resolved_via: probe/metadata/user
  group_by: [col]
  metrics:  [{agg, col, distinct:bool}]
  order_by, limit
  grain_note: str        # явная фиксация грануляции результата
  assumptions[]          # что агент предположил (показать юзеру)
  ambiguities[]          # нерешённые развилки
```

---

## 10. Observability (день 1)

Каждый узел и каждый вызов инструмента пишет трейс: вход, выход, использованный
контекст, probe-SQL и его стоимость, итоговый SQL с провенансом (какая колонка/
join откуда взялись). Без этого многошаговый LLM-агент неулучшаем и недебажим.
Формат — структурированный лог (jsonl) + человекочитаемый рендер сессии.

---

## 11. Governors (предохранители)

- `probe_sql`: только SELECT/EXPLAIN; авто-LIMIT; `statement_timeout`; **потолок
  cost по EXPLAIN** (отклонять дорогие сканы); запрет DML/DDL.
- Бюджет на вопрос: max шагов графа, max проб, max токенов. Превышение →
  graceful stop с объяснением, а не бесконечный цикл.
- Кэш результатов `describe_table`/`distinct_values`/проб в пределах сессии.
- Лимит итераций коррекции плана (анти-зацикливание «юзер↔агент»).

---

## 12. Адаптеры (переносимость)

- **LLM-адаптер:** реализован как провайдер-абстракция (`llm/`): `LLMBackend` +
  `LLMClient` (complete/complete_json с JSON-ремонтом). Бэкенды: `OpenAICompatBackend`
  (DeepSeek сейчас; осведомлён о reasoning-моделях — щедрый бюджет токенов, читаем
  `content`, не `reasoning_content`) и `GigaChatBackend` (прод, `langchain_gigachat`,
  модель `Gigachat-3-ultra`, глобальный rate-limit 5с + retry + повтор при
  finish_reason=blacklist). Выбор бэкенда — по `LLM_PROVIDER`. Узлы графа не знают
  провайдера. Низкая температура, маленькие промпты со схемой ответа.
- **Диалект-адаптер БД:** ILIKE, сэмплинг (`random()` vs TABLESAMPLE),
  формат/парсинг EXPLAIN, кавычки идентификаторов, особенности Greenplum
  (distribution keys влияют на fan-out/перф). Postgres-реализация сейчас,
  Greenplum-реализация — к проду.

---

## 13. Eval-харнес (важнейшее для уровня hermes/openclaw)

С первого коммита: набор кейсов `вопрос → {эталонный SQL | эталонный результат |
ожидаемые свойства}` + автогрейдер.

- **Грейдинг по результату**, а не по тексту SQL (несколько SQL валидны — см.
  кейс «февраль 2026»): сравнение множеств строк/значений на тестовой БД.
- Проверки-свойства: «нет размножения строк» (count до/после join), «выбрана
  ожидаемая таблица», «неоднозначность показана».
- Прогон на каждом изменении → измеримый ответ «стало ли лучше предыдущего».
- Seed: 3 эталона из §3, далее расширять.

## 14. Риски и принятые меры (честный реестр)

| Риск | Где бьёт | Мера в дизайне |
|---|---|---|
| Каталог не влезает в контекст | масштаб таблиц | `search_tables` retrieval-first (§5,§6) |
| Поиск join-пути — комбинаторика | масштаб | граф join-кандидатов оффлайн (§6) |
| `check_fanout` = full scan | масштаб строк | сначала кэш PK/unique, проба на сэмпле (§8) |
| PK по сэмплу неуникален/ложно-уникален | тихая ошибка | PK = гипотеза + сидирование из кураторской метадаты; живое подтверждение `check_join` (§6,§8) |
| Ошибки агрегации помимо дублей | тихая ошибка | узел `[grain&agg]` (§7,§8) |
| План окнули, SQL уехал | корректность | структурированный план-контракт + валидация (§9) |
| ILIKE на русском хрупкий | резолв значений | нормализация + «не найдено → уточнить» (§5) |
| Компаундинг ошибок слабой модели | надёжность | мелкие узлы + валидация/ремонт + бюджеты (§11,§12) |
| Диалект Greenplum ≠ Postgres | прод | диалект-адаптер (§12) |
| Нельзя измерить «лучше» | развитие | eval-харнес + трейсинг (§10,§13) |

## 15. Статус реализации

Собрано и проверено на живых данных (eval, 3 прогона/кейс):
- Q1 (count distinct) — 3/3; Q2 (join + анти-fanout, самый сложный) — 3/3
  стабильно; Q3 (неоднозначность) — ~5/6, выдаёт валидные трактовки.
- LLM-абстракция (DeepSeek↔GigaChat), диалект-граница (Postgres↔Greenplum),
  read-only guards, BM25-scope, join-граф, `check_join`, контракт плана,
  human-in-the-loop (interrupt + детерминированный `clarify`), коррекция плана,
  трассировка, eval-харнес.

## 16. Открытые вопросы / решено

- **Прод-СУБД Greenplum — подтверждено и подключаемо.** Адаптер переведён на
  SQLAlchemy `postgresql://` (psycopg2, Kerberos/GSSAPI), как боевой
  `core/database.py`. Коннект настраивается командой `/config_db_conn`
  (→ `config_db.json`); LLM переключается `/model` на GigaChat. Финальная выгрузка
  идёт через `run_export` (полный результат, без probe-LIMIT/cost-потолка).
  Остаётся переопределить адаптивный сэмплинг больших таблиц для `/refresh_metadata`.
- `search_tables` — стартуем с BM25 (без внешнего эмбеддера); при росте числа
  таблиц добавить векторный retrieval за тем же интерфейсом.
- Объёмы таблиц на проде — для стратегии сэмплинга проб и cost-потолков (`.env`).
- `last_query.csv` — сейчас UTF-8-SIG с заголовками (Excel/русский); подтвердить.
- Язык вопросов — пока русский; промпты на русском.

## 17. Известные ограничения

- **Детект неоднозначности флакует** (слабая модель): ~5/6. Кандидат на отдельный
  узел enumeration интерпретаций. Eval это измеряет.
- **Строковые value-фильтры не верифицируемы на тест-контейнере** (синтетика);
  проверяемы на реальных данных/проде.
- **Сложные SQL** (оконные функции, подзапросы) вне формы SPJA — нужен отдельный
  raw-SQL путь с обязательными `check_join`+EXPLAIN.
- Старые артефакты `data_for_agent/*.json` (rule_registry, semantic_lexicon,
  column/table_semantics) больше не используются каталогом — кандидаты на удаление.
