## 1. Вердикт (executive summary)

**Общая оценка: крепкий B+ / A− по замыслу, B− по инженерному исполнению, C+ по масштабируемости на прод.**

Что действительно хорошо и редко встречается в самодельных text2sql-системах:

- **Анти-fanout как центральный инвариант** — не декларация, а рабочая механика: join-граф с overlap
  значений, `check_join` живой пробой, авто-dedup через `DISTINCT ON`, детерминированный подбор ключа
  по PK справочника. Это самая сильная часть системы.
- **Контракт плана** (`StructuredPlan` → детерминированный рендер SQL) — правильная идея: пользователь
  окает структуру, SQL не может «уехать». Плюс human-in-the-loop через `interrupt` сделан корректно
  (детерминированный `clarify` отделён от недетерминированного LLM-вызова — это тонкость, которую
  большинство упускает).
- **Резолв значений фильтров живой пробой** (`resolve_values`) — значение берётся из БД, а не из
  устаревшей метадаты, с переключением на другую колонку. Это правильный паттерн.
- **Eval с первого дня**, грейдинг по свойствам результата, а не по тексту SQL. Многие «продукты»
  живут без этого годами.
- Отчёт/расследование строятся по правильному принципу: **числа считает pandas детерминированно,
  LLM — только смысловые решения** (роли колонок, подписи, нарратив). Для слабой модели это
  единственно верное распределение труда.

Что тянет вниз:

1. **Система срослась в два слабо связанных продукта** (SQL-агент и BI-генератор отчётов), которые
   не переиспользуют друг друга: отчёт грузит всю таблицу в pandas, вместо того чтобы пользоваться
   тем же SQL-конвейером; агент не умеет отвечать «аналитикой», отчёт не умеет SQL. На проде с
   Greenplum-таблицами в сотни миллионов строк `/build_business_report` без `--where` — это OOM
   ноутбука или 10-минутный таймаут.
2. **Мёртвый и легаси-код ~15–20% репозитория** (подробно в §4.3): целый спек-конвейер отчёта,
   примитивы-графики, `investigate._drill`, корневой `metadata_refresh.py`, импортирующий
   несуществующий пакет `core.*`, неиспользуемые конфиги.
3. **Защитный слой written in regex**: SQL-guard, детект «денег», детект «причин», классификация
   колонок, распознавание выражений в рендере — всё на регулярках. Это работает на четырёх таблицах
   демо-схемы и начнёт стрелять на первой же нестандартной таблице. Парсер SQL (sqlglot) решил бы
   половину этих проблем строго.
4. **Прод-риски GigaChat не закрыты**: `max_tokens` в GigaChat-бэкенде игнорируется, глобальный
   rate-limit 5 с × 6–9 последовательных LLM-вызовов на отчёт = минуты ожидания на ровном месте,
   worst-case цепочка ретраев JSON — десятки вызовов. Нет function calling, хотя GigaChat его
   поддерживает — JSON добывается «уговорами и ремонтом».
5. **Нет памяти и нет диалога**: каждый вопрос — новый thread, коррекции живут только внутри одного
   плана. «А теперь то же самое, но за март» не работает. Одобренные планы никуда не сохраняются
   и не переиспользуются как few-shots — система не учится.
6. **Ноль юнит-тестов, нет пакетирования** (`sys.path.insert` вместо `pyproject.toml`), деплой —
   «скопируйте папку», нет CI. Для S-tier это блокер.

## 2. Карта системы (как есть)

```
CLI (Jupyter REPL, блокирующий input())
 ├── Agent (LangGraph, MemorySaver, thread на вопрос)
 │     scope → [clarify] → plan_build → resolve_values → join_advisor
 │           → present (interrupt) → synth → execute → last_query.csv
 │     инструменты: search_tables(BM25) / describe / distinct_values / probe_sql / check_join
 ├── /build_business_report → report/builder.py
 │     _load(вся таблица в pandas) → labels(LLM) → roles(LLM) → behaviors(LLM)
 │     → mining (детекторы: rate_dev, interaction 2D, money_conc, value_mismatch)
 │     → patterns (сезонность, повторяемость, всплески) → narrate(LLM) → MD+HTML
 ├── /investigate → report/investigate.py
 │     _prepare (те же labels/roles/behaviors) → frame(LLM) → decompose/tree/waterfall/
 │     pareto/why/who/quadrant (pandas) → synthesize(LLM) → MD+HTML
 └── /refresh_metadata → catalog/refresh.py
       introspect + адаптивный сэмпл + PK-перебор + join-overlap + LLM-описания
       → tables_list.csv / attr_list.csv / join_candidates.json
```

Слои: `llm/` (OpenAI-compat + GigaChat за общим `LLMClient` с JSON-ремонтом), `db/`
(SQLAlchemy + regex-guard + cost-потолок), `catalog/`, `plan/` (модель + рендер),
`trace.py` (jsonl), eval-харнес на 7 кейсов.

## 3. Конкретные ошибки и баги

Отсортированы по серьёзности. Формат: **[критичность] файл:строка — суть**.

### 3.1 Критичные (уронят или заблокируют прод)

**B-1. `PROBE_TIMEOUT_MS` — мёртвый конфиг; пробы работают с таймаутом выгрузки (10 минут).**
`config.py:36` объявляет `probe_timeout_ms=15000`, но он **нигде не используется**. Engine создаётся
один раз в `db/adapter.py:156-161` с `statement_timeout={export_timeout_ms}` (600 000 мс) для *всех*
соединений. То есть каждая «лёгкая проба» агента (`check_join`, `distinct_values`, overlap при
рефреше) может висеть 10 минут на большой Greenplum-таблице. README и DESIGN обещают
«timeout» проб — обещание не выполнено. Это главный источник будущих «агент завис».

**B-2. `key_unique` — full scan `GROUP BY ... HAVING count(*)>1`, а не проба на сэмпле.**
`tools/toolbox.py:111-123`. DESIGN §8 прямо требует «живая проба на СЭМПЛЕ, не full scan», риск
№3 реестра рисков. По факту — полная агрегация по таблице. На проде либо (а) упрётся в
`probe_max_cost` → `GuardError`, который **не перехватывается** в `join_advisor`
(`nodes.py:269` — `check_join` зовётся без try/except) → весь `ask()` падает исключением, либо
(б) при щедром потолке — минуты ожидания на каждый join × 2 стороны × перепроверки. Любой
вопрос с join к крупной прод-таблице сейчас или падает, или тормозит.

**B-3. `/build_business_report` и `/investigate` без `--where` грузят ВСЮ таблицу в pandas.**
`report/builder.py:33-39` — `run_export(sql, enforce_limit=False)`, ни оценки размера
(`estimate_row_count` есть в адаптере и не используется здесь!), ни подтверждения, ни
автосэмплирования. На демо-контейнере 4–6 тыс. строк — норм; на проде `uzp_dwh_sale_funnel_task` —
это десятки миллионов строк в память Jupyter-пода. Предсказуемый инцидент №1 после деплоя.

**B-4. GigaChat-бэкенд игнорирует `max_tokens`.**
`llm/gigachat.py:67-88` — параметр принят и выброшен: в `GigaChat(...)` и `llm.invoke(messages)`
бюджет токенов не передаётся. Все тщательно подобранные бюджеты узлов (`max_tokens=8000` для
профайлера, 3500 для плана…) на проде не действуют. Длинный ответ обрежется дефолтом модели —
и JSON-парсинг начнёт флакать именно там, где его лечили.

**B-5. Worst-case каскад ретраев: до ~50 LLM-вызовов и ~10+ минут на один `complete_json`.**
`llm/client.py`: `complete_json` = 3 попытки × (генерация + ремонт), каждая из которых — `complete`
с лестницей из 8 попыток (4 быстрых + 3 медленных по 30 с). Под GigaChat каждый вызов ещё ждёт
глобальные 5 с (`MIN_INTERVAL`) и имеет **свои** 5 ретраев внутри бэкенда. Слои ретраев
перемножаются. Нужен единый бюджет ретраев на запрос, а не три независимых слоя.

### 3.2 Серьёзные (неверное поведение / тихая деградация)

**B-6. SQL-guard режет валидные запросы и основан на подстроках.**
`db/adapter.py:37-42, 182-189`: `_FORBIDDEN` ищет слова `set|do|call|comment|copy|merge|begin...`
как `\b`-токены **в любом месте строки, включая строковые литералы и имена колонок**.
`WHERE task_subtype ILIKE '%созда%'` — ок, но `WHERE comment_text ILIKE '%do%'`... ладно, `\b` в
имени `comment_text` не матчится, а вот литерал `'%обновление (update)%'` — отклоняется; колонка
с именем `set_id`/`do_flag` в выражении — отклоняется частично (`\bset\b` не матчится в `set_id`,
но `merge`/`call`/`do` как отдельные слова в русско-английских литералах — легко). Плюс `;` внутри
строкового литерала → ложный отказ «несколько операторов». Raw-SQL-режим (`plan_build` → `raw_sql`)
делает guard основной линией обороны — а он дырявый в обе стороны: `SELECT pg_sleep(600)` или
тяжёлый cross join проходят (read-only ≠ безопасно), а безобидный литерал — нет.

**B-7. Выражения в плане рендерятся вербатим — контракт «SQL не уедет от плана» неполный.**
`plan/render.py:22-27, 40-47`: всё, что `_EXPR_RE` посчитал «выражением» (скобка, пробел, `/`, `*`),
вставляется в SQL как есть — включая подзапросы `(SELECT ...)`, которые guard пропустит. Модель
может (не по злому умыслу, а по глюку) положить в `projections.column` коррелированный подзапрос,
и он выполнится, хотя пользователь окнул «план» с нечитаемой строкой. Раз уж выражения разрешены —
их надо парсить (sqlglot) и валидировать по AST-белому списку функций, а не по regex.

**B-8. `resolve_values` ослепляет фильтры: `=` всегда превращается в `ILIKE '%...%'`.**
`nodes.py:204-209`: если значение нашлось пробой, оператор принудительно становится `ILIKE`
с обрамлением `%…%`. Для точного enum-значения (`status = 'Закрыта'`) это меняет семантику:
`'%Закрыта%'` зацепит «Не закрыта» — классический баг подстроки. `_best_match` «самое короткое
совпадение» — та же ловушка наоборот.

**B-9. Комментарии-описания сидируются по имени колонки глобально, без привязки к таблице.**
`catalog/refresh.py:152-174` (`_merge_desc_seed_from_csv`, `_col_seed` ключуется `name.lower()`):
колонка `status` из таблицы задач получит описание колонки `status` из любой другой таблицы.
Плюс **самоподдерживающийся цикл**: LLM-сгенерированное описание при следующем рефреше становится
«кураторским сидом» с приоритетом над свежим LLM — ошибка, однажды сгенерированная, фиксируется
навсегда.

**B-10. `_find_pk` — комбинаторный взрыв на широких таблицах.**
`catalog/refresh.py:77-96`: перебор `combinations(cols, size)` до size=4. На таблице в 60–100
колонок это миллионы комбинаций × `df.duplicated()` по 100k-сэмплу каждая — рефреш зависнет без
всякого таймаута (per-table timeout защищает только SQL-сэмпл, не клиентский перебор).

**B-11. Overlap-пробы рефреша тихо деградируют join-граф.**
`catalog/refresh.py:496-509`: `_overlap` при любой ошибке (включая cost-потолок на больших
таблицах) возвращает `0` → пара выбрасывается. На проде join-граф может молча получиться пустым,
и вся анти-fanout-механика (`suggest_join`) деградирует в «нет кандидатов», о чём никто не узнает.

**B-12. `GreenplumAdapter` — пустой класс, а различия реальны.**
`db/adapter.py:343-345`. `ORDER BY random()` на GP-таблице в миллиард строк, cost-модель GP
(другие масштабы Total Cost → один `probe_max_cost` на оба диалекта бессмыслен), `DISTINCT ON`
+ `ORDER BY` поверх distribution key, отсутствие нормального `EXPLAIN (FORMAT JSON)` в старых GP
(GP6 = PG 9.4 — формат есть, но структура плана со slice/motion отличается, парсинг
`data[0]["Plan"]["Total Cost"]` может не совпасть). «Диалект-граница» задекларирована, но не
реализована.

**B-13. Трассировка пишет полные промпты и данные на диск без ограничений.**
`trace.py` (все события с `system`/`user`/`text`), `logging_setup.py` (DEBUG-лог всего),
`nodes.py:400` (первые строки результата в `agent.log`). Для банковских данных (ИНН, ФИО, ЗП)
это несанкционированное копирование чувствительных данных в файлы `traces/*.jsonl` и
`workspace/agent.log`, которые README предлагает «прислать при сбое». Нужна редакция/маскирование
и ротация (сейчас `FileHandler` без ротации — лог растёт бесконечно).

**B-14. `root.handlers.clear()` уничтожает логирование хост-приложения.**
`logging_setup.py:28`. Библиотека, сносящая root-логгер процесса, в Jupyter с другими
расширениями — источник загадочных «у меня пропали логи».

скрипты, требующие живую БД и LLM).

## 4. Архитектурные слабости по подсистемам

### 4.1 Text2SQL-агент

1. **Граф выродился относительно DESIGN.** Задумка «короткий контекст, много мелких шагов»
   (`scope → explore → grain&agg → join_advisor → plan → validate`) на деле схлопнулась:
   узлов `explore` и `grain&agg` нет, `validate` растворён в `synth`. Вместо этого —
   **один гигантский PLAN_SYS на ~70 строк правил** (`graph/prompts.py:41-108`), где смешаны:
   анти-fanout, dedup, полуоткрытые интервалы, ILIKE-резолв, HAVING, выражения, окна, raw-SQL.
   Это прямо противоречит собственному принципу проекта и является худшим режимом для слабой
   модели: один вызов должен удержать в голове 15 правил сразу. Половина последующих узлов
   (`_correct_count_ids`, `resolve_values`, `join_advisor`) — это фактически **заплатки,
   исправляющие ошибки перегруженного плана**, вместо того чтобы не дать их совершить.

2. **Нет диалога.** `Agent.ask` открывает новый thread (`agent.py:121-126`), контекст
   предыдущего вопроса теряется. «Теперь добавь разрез по сегменту» после выполненного запроса —
   начинается с нуля, включая повторные пробы БД. Follow-up — базовое ожидание пользователя
   от такого инструмента.

3. **Нет накопления знаний.** Каждая коррекция пользователя («это поле не деньги», «джойни по
   old_gosb_id») исчезает после сессии. Одобренные (человеком!) пары вопрос→план — это золотой
   датасет few-shots, который никто не собирает, хотя трейсы уже пишутся.

4. **Коррекция перезапускает весь конвейер.** `present → plan_build` с добавленной строкой
   коррекции — план строится заново целиком (state сохраняет только corrections). Заявленное в
   DESIGN «точечный re-run затронутого узла» не реализовано. Для мелкой правки («LIMIT 10»)
   это лишние 2–4 LLM-вызова и повторные пробы.

5. **`MemorySaver`** — состояние живёт в памяти ядра; перезапуск ноутбука убивает сессию.
   LangGraph умеет `SqliteSaver` из коробки — персистентность стоила бы 3 строки.

6. **Ambiguity-опции нельзя комбинировать** и нельзя ответить свободным текстом с уточнением —
   `_resolve_ambiguity` понимает только индекс/подстроку label.

7. **`recency_col` и авто-dedup содержат доменные русизмы** («карточк», «созда») —
   `toolbox.py:204-218`. Мелочь, но это ровно тот «хардкод под таблицы», который DESIGN запрещает.

### 4.2 LLM-слой

1. **Не используется function calling GigaChat.** GigaChat-3 поддерживает функции/структурные
   ответы через `langchain_gigachat` (`bind_tools` / `with_structured_output`). Вместо этого —
   «Ответь ТОЛЬКО валидным JSON…» + четырёхступенчатый самодельный ремонт (`_clean_json`,
   `_extract` со счётчиком скобок). Это главный источник хрупкости и лишних токенов. Со
   структурным выводом исчезает целый класс ретраев.

2. **Один системный + один user-месседж на вызов** — интерфейс `chat(system, user)` не позволяет
   ни few-shots как отдельных сообщений, ни продолжения диалога, ни префилла. Для слабой модели
   few-shot-примеры в роли assistant заметно стабильнее «схемы ответа в тексте».

3. **Rate-limit 5 с — константа класса**, не конфиг; usage/токены не учитываются (`LLMResult.usage`
   пуст для GigaChat) — стоимость и деградацию нельзя мониторить.

4. **Ретраи слоятся** (см. B-5) и `sleep` держится внутри глобального лока (`gigachat.py:61-65`) —
   при будущей параллелизации это станет узким местом.

### 4.3 DB-слой

1. Guard должен стать **парсером, а не regex** (sqlglot: один `parse_one`, проверка типа
   statement, обход AST на запрещённые узлы, автодобавление LIMIT через AST — и сразу решаются
   B-6, B-7, `_apply_limit`-обёртка и подсчёт «одного оператора»).
2. `run_select` делает **EXPLAIN перед каждой пробой** — два раундтрипа; на GP EXPLAIN сам по
   себе не бесплатен. Кэш планов/costs по нормализованному SQL напрашивается.
3. `estimate_row_count`, `metadata_sample` есть — но не используются нигде, кроме рефреша.
   Отчёт (B-3) обязан ими пользоваться.
4. Пул соединений один, `statement_timeout` один на всё (B-1): нужно два профиля соединений —
   probe (короткий таймаут, `work_mem` поменьше) и export.

### 4.4 Каталог и метаданные

1. **CSV как хранилище** уже трещит: `|`-разделители (B-21), полная перезапись при каждом
   рефреше, никакого версионирования/istoрии. SQLite (или DuckDB) файл решил бы всё и позволил
   бы инкрементальный рефреш и историю изменений описаний.
2. **PK-гипотеза** ищется перебором на сэмпле (B-10) — вместо этого стоит спрашивать статистику
   БД (`pg_stats.n_distinct`, GP: `gp_distribution_policy` — distribution key почти всегда
   входит в бизнес-ключ) и только потом добирать перебором по кандидатам с высокой уникальностью.
3. **Семантическая классификация** (`_classify`) — регекс по имени. LLM уже вызывается для
   описаний — та же итерация может выдавать и класс, и грануляцию, и роль таблицы, и связи
   (обогащённый каталог за один батч-вызов на таблицу).
4. **Синтетические сэмплы генерируются на проде по умолчанию** (`gen_samples = bool(llm)`) —
   лишние тысячи токенов и минуты на каждую таблицу рефреша ради файла, нужного только для
   тестового контейнера.

### 4.5 CLI/UX

1. **Jupyter-REPL с блокирующим `input()`** — тупик: нет истории, нет автодополнения таблиц,
   нет прогресса LLM-вызовов (только `…статус`), невозможно прервать долгий шаг, вывод плана —
   plain text. Даже оставаясь в ноутбуке, `ipywidgets`-чат дал бы кликабельные варианты
   неоднозначности, кнопку «выполнить», рендер плана таблицей. Следующая ступень — тонкий
   web-UI (см. §6).
2. Парсинг команд — цепочка `elif` на 60 строк; `--where` вычленяется вручную. Мини-роутер
   команд с декларацией аргументов упростил бы и help, и валидацию.
3. Результат показывается как `df.head(20)` без итогов/форматирования чисел; SQL — после
   результата, а не до.

### 4.6 Бизнес-отчёт

1. **Пять последовательных LLM-вызовов** (labels → roles → behaviors → [focus] → narrate) с
   пересекающимися входами (одни и те же колонки+описания+примеры). Под 5-секундный rate-limit
   GigaChat это ≥25–45 с чистого ожидания. Всё, кроме narrate, — **один** структурный вызов
   «профиль таблицы» (labels+units+roles+primary+behaviors+record_count в одной схеме) — и
   быстрее, и согласованнее (не будет расхождений «профайлер сказал metric, labels сказал other»).
2. **Результат профилирования не кэшируется.** Повторный отчёт по той же таблице заново
   спрашивает LLM обо всём. Профиль таблицы — это метаданные; его место — в каталоге (обновлять
   при рефреше), а не в рантайме отчёта.
3. **Пороговые константы рассыпаны по коду** (0.05/1.3/1.43/0.25/1.8/0.08/4×...) без единого
   конфига чувствительности. Настроить «меньше находок / больше находок» нельзя.
4. Секции отчёта не имеют сквозной «важности» — порядок задан типами, а не score; главная
   находка может оказаться в середине.
5. `narrate` получает только `facts` (без таблиц) — норм, но не получает **связи между
   секциями** (напр. один и тот же ГОСБ фигурирует в трёх находках) — сквозной сюжет LLM
   собрать не из чего.

### 4.7 Расследования

Концептуально это лучший модуль (frame → decompose → tree → why → who → synth — честная
методология аналитика). Слабости:

1. **Один уровень «где»**: несмотря на наличие (мёртвой) функции спуска `_drill`, реальное
   дерево — только `primary_dim → entity`. «Сегмент → территория → холдинг → компания» не
   строится, хотя `_decompose` это умеет. Мёртвый `_drill` это и был — его нужно вернуть в строй.
2. **«Почему» = преобладающее значение разреза-причины.** Нет сравнения с базой: если 80%
   всех строк вообще имеют причину «Иное», то «преобладает Иное (80%)» — не инсайт. Нужен lift
   против общего распределения (в mining это уже сделано — `_rate_deviations`! — но в
   investigate не переиспользовано).
3. **Нет времени.** Вопрос «где потеряли к 2026» напрашивается на «когда именно» — динамика
   потерь по месяцам с детекцией перелома отсутствует (в mining есть `spikes`, здесь нет).
4. Дублирование `_prepare` с билдером (те же 3 LLM-вызова заново) — общий «профиль таблицы»
   решил бы и это.
5. `_frame` fallback `num_cols[0]` — произвольная колонка как target при промахе LLM: лучше
   честный отказ с просьбой уточнить.

### 4.8 Eval и инженерная культура

1. 7 SQL-кейсов, ни одного негативного (инъекции/guard), ни одного на dedup-корректность
   значений (сумма после dedup = эталон), ни одного на «отказ» (вопрос вне данных). Отчёт
   проверяется структурно («есть 4 секции»), но не по содержанию (правильные ли числа в KPI —
   легко проверить пандасом против эталонного сэмпла).
2. Нет юнит-тестов на чистые функции, которых много и которые идеально тестируемы:
   `_normalize_on`, `normalize_plan`, `render_sql`, `_apply_limit`, `_find_pk`, `_decompose`,
   `_select`, `fmt_val`, guard. Это дешёвые сотни ассертов, ловящие регрессии без LLM.
3. Нет фиксации LLM-ответов (VCR-кассеты/golden-файлы) — eval недетерминирован и требует живой
   модели даже для проверки кода вокруг неё.

## 5. Где решения неуникальны (и что мир делает лучше)

| Место в харнесе | Самодельное решение | State of the art |
|---|---|---|
| SQL-guard, авто-LIMIT | regex-подстроки | **sqlglot**: AST-валидация, тип стейтмента, трансформации (LIMIT, квотинг), транспиляция Postgres↔Greenplum |
| JSON от LLM | «ответь только JSON» + ручной ремонт | **function calling / structured output GigaChat**, pydantic-схема как контракт |
| Retrieval таблиц | BM25 без морфологии | гибрид: эмбеддинги (мультиязычные bge-m3/e5 локально) + BM25 со стеммингом + rerank; для <50 таблиц хватит стемминг+синонимы из каталога |
| Выбор колонок/значений | весь стол колонок в один промпт | **schema linking** как отдельный шаг (выбор релевантных колонок до планирования) — стандарт всех топ-решений BIRD/Spider |
| Генерация SQL | один план с первого раза + заплатки | **multi-candidate + execution-based selection**: 2–3 кандидата (структурный план и raw-SQL), прогон EXPLAIN/пробы, самосогласованность результатов — стандарт (CHASE-SQL, XiYan-SQL и пр.) |
| Ошибки исполнения | goto error-конец | **self-correction loop**: текст ошибки БД → обратно модели с планом → повторный синтез (1–2 итерации). У вас это есть только для JSON и fanout, но не для SQL-ошибок |
| Метрики/семантика | регексы «деньги/люди» в трёх местах | **semantic layer / metric store** (лёгкий YAML: метрика = выражение + единица + агрегат + разрешённые разрезы), заполняется LLM-ом один раз, правится человеком, используется и агентом, и отчётами |
| Память | нет | хранение одобренных пар Q→план как few-shots с retrieval по похожести вопроса (self-improving loop без дообучения — то, что нужно при фиксированной gigachat-3-ultra) |
| Оценка | скрипт с print | pytest + golden-кассеты + матрица pass-rate в CI |
| Отчёт | pandas на всей таблице | **push-down в SQL**: все groupby/agg майнинга выражаются в 5–10 GROUP BY-запросов, которые GP выполняет за секунды на миллиардах строк; в pandas — только результаты |

Отдельно: **сам форм-фактор отчёта** (статический HTML с base64-PNG) устарел относительно
доступного при тех же затратах: интерактивные Plotly/ECharts-графики в самодостаточном HTML
(тултипы, зум, сортировка таблиц) генерируются так же из pandas и радикально повышают
воспринимаемую ценность.

## 6. Как сделать в разы лучше: целевая S-tier архитектура

Ограничение одно — `gigachat-3-ultra`. Значит выигрывать нужно не «моделью поумнее», а
**структурой вокруг модели**: меньше решений на вызов, структурный вывод, исполнение как
арбитр, память как усилитель. Ниже — целевая картина.

### 6.1 Ядро: единый «семантический слой» вместо трёх систем метаданных

Сейчас знание о данных размазано: каталог CSV, LLM-профиль отчёта (labels/roles/behaviors),
frame расследования, регексы «деньги/причины». Целевое состояние — **один артефакт
`semantic_model` на таблицу** (SQLite/YAML в `data_for_agent`), заполняемый при
`/refresh_metadata` одним батч-вызовом LLM и правкой человека:

```yaml
table: uzp_dwh_fact_outflow
grain: "одно событие оттока (gosb_id, inn, report_dt)"
role: event
time_column: report_dt
entities:   [{col: inn, label: "ИНН компании", name_col: company_name}]
dimensions: [{col: segment_name, label: "Сегмент", card: 5}]
measures:
  - {name: outflow_qty, label: "Отток, чел.", agg: sum, unit: people}
  - {name: outflow_fot_amt, label: "Отток ФОТ", agg: sum, unit: money}
derived:
  - {name: overdue_rate, label: "Просрочка закрытия", type: overdue, plan: plan_dttm, fact: fact_dttm}
joins:
  - {to: uzp_dim_gosb, on: [[tb_id, tb_id],[gosb_id, old_gosb_id]], verified: "2026-07-01", card: "N:1"}
reasons: [outflow_reason_name]
scope_rules: [{measure: deal_potential, valid_when: "task_category='Предложение'"}]
```

Эффекты каскадом:
- text2sql-агент получает готовые проверенные join-пути (не гипотезы) и единицы измерения;
- отчёт и расследование **перестают профилировать таблицу на каждый запуск** (минус 3 LLM-вызова
  и минус рассогласования);
- «деньги/не деньги» решается один раз и правится человеком (ваш `money_confirm`-диалог
  становится редактором семантической модели, а не одноразовым вопросом);
- eval может проверять семантическую модель отдельно от агента.

### 6.2 Text2SQL: конвейер «мало решений на вызов» + исполнение как арбитр

```
Q → context builder (retrieval: таблицы + релевантные КОЛОНКИ + похожие прошлые Q→план)
  → schema linking (LLM-вызов №1: выбрать колонки/фильтры/интерпретации; surfaced ambiguity)
  → plan synthesis (LLM-вызов №2: StructuredPlan НА УЖЕ ОТОБРАННЫХ колонках; короткий промпт)
  → deterministic hardening (resolve_values c сохранением '=', join из semantic_model,
    normalize, sqlglot-валидация, EXPLAIN)
  → cheap execution probe (LIMIT 100) → если ошибка/пусто → self-correction (LLM-вызов №3
    с текстом ошибки; максимум 2 итерации)
  → present (план + превью 5 строк результата!) → ok → полный экспорт
```

Ключевые отличия от текущего:
1. **Schema linking отдельно от плана** — режет главный промпт вдвое и убирает заплатки
   `_correct_count_ids` (выбор идентификатора — это schema linking).
2. **Превью результата до апрува.** Сейчас пользователь окает SQL вслепую. Проба LIMIT 100 стоит
   копейки и ловит 80% семантических ошибок («пусто», «дубли», «не те единицы») ДО полного
   исполнения. Это самое дешёвое улучшение качества во всей системе.
3. **Self-correction по ошибкам БД** — сейчас любой EXPLAIN-fail = конец сессии.
4. **Память**: одобренный план сохраняется `{question, plan, tables}`; при новом вопросе
   top-2 похожих подмешиваются few-shots. С фиксированной моделью это единственный механизм
   «система становится умнее со временем» — и он почти бесплатный (трейсы уже есть).
5. **Диалог**: thread не сбрасывается; предыдущий план в state; «а теперь по месяцам» — это
   маленький LLM-вызов «модифицируй план», а не полный цикл.
6. Для сложных запросов — **два кандидата** (структурный план + raw-SQL) с выбором по
   исполнению на пробе: у слабой модели self-consistency даёт больший прирост, чем любой
   промпт-инжиниринг.

### 6.3 Отчёты и расследования: push-down + интерактив

1. **SQL push-down.** Все агрегаты майнинга (`groupby(d)[m].agg`) выражаются в GROUP BY.
   Схема: один проход «профиль» (счётчики, кардинальности) → 6–12 агрегирующих запросов
   (по одному на разрез×меру, GP их щёлкает параллельно) → pandas работает с результатами
   ×1000 меньшего размера. Отчёт по таблице в 500 млн строк становится минутным делом без
   `--where`. `estimate_row_count` решает, можно ли грузить сырьё (< N строк — как сейчас,
   иначе push-down).
2. **Один вызов «профиль таблицы»** вместо labels+roles+behaviors (см. §6.1) + один вызов
   narrate/synth. Итого 2 LLM-вызова на отчёт вместо 5–6: под 5-секундный rate-limit — это
   разница между 15 и 45 секундами.
3. **Интерактивный HTML** (Plotly в self-contained файле): зум, тултипы с точными числами,
   сортируемые таблицы, drill-down кликом по бару в «кто внутри». Расследование с
   кликабельным treemap — другой продукт по ощущению.
4. **Investigate: вернуть многоуровневый спуск** (реанимировать `_drill`), добавить
   временной срез («когда началось») и lift-нормировку «почему» (переиспользовать
   `_rate_deviations`).
5. **Investigate поверх SQL-агента**: `/investigate` на вопрос «почему упал X в марте»
   должен уметь сам сгенерировать `--where` и выбрать таблицу через тот же scope-механизм —
   сейчас пользователь обязан знать таблицу и синтаксис фильтра.

### 6.4 Платформа

1. **LLM-слой**: перейти на структурный вывод GigaChat (`with_structured_output`/functions);
   единый бюджет ретраев; passthrough `max_tokens`; учёт usage; конфигурируемый rate-limit;
   кэш вызовов по хэшу (system,user) — рефреш метаданных и повторные отчёты станут почти
   бесплатными.
2. **DB-слой**: sqlglot-guard; два профиля соединений (probe 15 с / export 600 с — починить B-1);
   `key_unique` на сэмпле (`WHERE random()<p` подзапрос) с эскалацией до полного скана только
   по требованию; реальный `GreenplumAdapter` (cost-нормировка, distribution keys в
   join-советнике).
3. **Хранилище**: каталог+семантическая модель+память планов в одном SQLite; атомарные записи;
   trace с маскированием значений (хэш вместо сырых данных) и ротацией.
4. **Пакетирование**: `pyproject.toml`, `pip install -e .`, extras `[prod]`/`[dev]`; убрать
   `sys.path.insert`; вычистить весь мёртвый код (§B-15); CI: ruff + mypy + pytest (юниты без
   LLM) + nightly eval с кассетами.
5. **UI**: ближний шаг — ipywidgets-чат в ноутбуке (кликабельные опции, кнопки ok/править,
   прогресс шагов); целевой — тонкий FastAPI+HTMX/React чат с историей сессий, живым стримом
   статусов графа (LangGraph это отдаёт событиями), таблицей результата и кнопкой «отчёт по
   этому результату» — смычка двух половин продукта.

### 6.5 Eval как двигатель (а не приложение)

1. Юнит-слой: чистые функции (см. §4.8) — сотни быстрых тестов.
2. Кассеты LLM-ответов для детерминированного прогона конвейера без модели.
3. Расширить кейсы: негативные (guard, инъекции в фильтры), числовая корректность dedup-join,
   отказы («данных нет»), диалоговые (follow-up), отчётные с эталонными числами KPI.
4. Метрики в CI: pass-rate по кейсам, латентность на кейс, число LLM-вызовов на кейс
   (регрессия эффективности — тоже регрессия).

### Финальное слово

Фундамент выбран верно: инварианты вместо «магии», человек в контуре, детерминированные числа,
LLM только там, где нужен смысл. Именно поэтому у системы есть реальный путь к S-tier: не
переписывание идеи, а (1) выправление прод-рисков B-1…B-5, (2) консолидация трёх систем
метазнания в одну семантическую модель, (3) переход guard/рендера на AST, (4) исполнение как
арбитр качества (превью + self-correction + кандидаты) и (5) память как компенсатор фиксированной
модели. Каждый из этих шагов усиливает остальные — и все они достижимы на gigachat-3-ultra,
потому что уменьшают, а не увеличивают требования к интеллекту модели.
