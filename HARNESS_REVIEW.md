# Аудит харнеса Text2SQL — заключение по агенту

> Полный разбор кодовой базы **самого агента**: text2sql-граф (LangGraph), LLM/DB-слои,
> каталог метаданных, CLI, eval, инженерная платформа. Прочитан каждый файл в `src/`,
> `eval/`, `scripts/`, корневые скрипты и данные.
>
> **Разделение документов:** всё, что касается аналитических функций
> (`/build_business_report`, `/investigate`, качество отчётов, дэши, плейбуки/impact) —
> вынесено в **`ANALYTICS_CORE_DESIGN.md`** (диагноз + целевой дизайн + приложение с
> перенесёнными находками аудита). Здесь такие находки оставлены короткими ссылками
> с сохранением сквозной нумерации B-*.
>
> **Порядок реализации (решение владельца):** сначала аналитическое ядро
> (`ANALYTICS_CORE_DESIGN.md`), затем перестройка text2sql. Дорожная карта §7 разбита
> на очереди 0/1/2 с учётом зависимостей между ними.
>
> Формат: вердикт → конкретные баги → архитектурные слабости по подсистемам →
> где решения неуникальны → как сделать S-tier (единственное ограничение — модель
> `gigachat-3-ultra`). Документ только описывает — код не менялся.

## 1. Вердикт (executive summary)

**Общая оценка агента: крепкий B+ / A− по замыслу, B− по инженерному исполнению,
C+ по готовности к проду.**

Что действительно хорошо и редко встречается в самодельных text2sql-системах:

- **Анти-fanout как центральный инвариант** — не декларация, а рабочая механика: join-граф
  с overlap значений, `check_join` живой пробой, авто-dedup через `DISTINCT ON`,
  детерминированный подбор ключа по PK справочника. Это самая сильная часть системы.
- **Контракт плана** (`StructuredPlan` → детерминированный рендер SQL) — правильная идея:
  пользователь окает структуру, SQL не может «уехать». Плюс human-in-the-loop через
  `interrupt` сделан корректно (детерминированный `clarify` отделён от недетерминированного
  LLM-вызова — тонкость, которую большинство упускает).
- **Резолв значений фильтров живой пробой** (`resolve_values`) — значение берётся из БД,
  а не из устаревшей метадаты, с переключением на другую колонку. Правильный паттерн.
- **Eval с первого дня**, грейдинг по свойствам результата, а не по тексту SQL.

Что тянет вниз:

1. **Система срослась в два слабо связанных продукта** (SQL-агент и BI-генератор), которые
   не переиспользуют друг друга. Разбор этой проблемы и целевой дизайн объединения —
   полностью в `ANALYTICS_CORE_DESIGN.md`; здесь дальше — только агентная половина.
2. **Мёртвый и легаси-код** (§B-15): в агентной части — корневой `metadata_refresh.py`
   на 1040 строк, импортирующий несуществующий пакет `core.*`, неиспользуемые конфиги,
   бэкап-артефакты; плюс мёртвые аналитические модули (перечислены в
   `ANALYTICS_CORE_DESIGN.md`, приложение).
3. **Защитный слой written in regex**: SQL-guard, распознавание выражений в рендере,
   классификация колонок каталога — всё на регулярках. Работает на четырёх таблицах
   демо-схемы и начнёт стрелять на первой же нестандартной. Парсер SQL (sqlglot) решил
   бы половину проблем строго.
4. **Прод-риски GigaChat не закрыты**: `max_tokens` игнорируется бэкендом, глобальный
   rate-limit 5 с, worst-case цепочка ретраев JSON — десятки вызовов. Нет function
   calling, хотя GigaChat его поддерживает — JSON добывается «уговорами и ремонтом».
5. **Нет памяти и нет диалога**: каждый вопрос — новый thread, коррекции живут только
   внутри одного плана. «А теперь то же самое, но за март» не работает. Одобренные планы
   не сохраняются и не переиспользуются как few-shots — система не учится.
6. **Ноль юнит-тестов, нет пакетирования** (`sys.path.insert` вместо `pyproject.toml`),
   деплой — «скопируйте папку», нет CI. Для S-tier это блокер.

## 2. Карта системы (как есть)

```
CLI (Jupyter REPL, блокирующий input())
 ├── Agent (LangGraph, MemorySaver, thread на вопрос)
 │     scope → [clarify] → plan_build → resolve_values → join_advisor
 │           → present (interrupt) → synth → execute → last_query.csv
 │     инструменты: search_tables(BM25) / describe / distinct_values / probe_sql / check_join
 ├── /build_business_report, /investigate            → см. ANALYTICS_CORE_DESIGN.md
 └── /refresh_metadata → catalog/refresh.py
       introspect + адаптивный сэмпл + PK-перебор + join-overlap + LLM-описания
       → tables_list.csv / attr_list.csv / join_candidates.json
```

Слои: `llm/` (OpenAI-compat + GigaChat за общим `LLMClient` с JSON-ремонтом), `db/`
(SQLAlchemy + regex-guard + cost-потолок), `catalog/`, `plan/` (модель + рендер),
`trace.py` (jsonl), eval-харнес на 7 кейсов.

## 3. Конкретные ошибки и баги

Отсортированы по серьёзности. Формат: **[критичность] файл:строка — суть**.
Нумерация B-* сквозная; находки по аналитическим функциям перенесены в
`ANALYTICS_CORE_DESIGN.md` (приложение) с сохранением номеров.

### 3.1 Критичные (уронят или заблокируют прод)

**B-1. `PROBE_TIMEOUT_MS` — мёртвый конфиг; пробы работают с таймаутом выгрузки (10 минут).**
`config.py:36` объявляет `probe_timeout_ms=15000`, но он **нигде не используется**. Engine
создаётся один раз в `db/adapter.py:156-161` с `statement_timeout={export_timeout_ms}`
(600 000 мс) для *всех* соединений. То есть каждая «лёгкая проба» агента (`check_join`,
`distinct_values`, overlap при рефреше) может висеть 10 минут на большой Greenplum-таблице.
README и DESIGN обещают «timeout» проб — обещание не выполнено. Главный источник будущих
«агент завис».

**B-2. `key_unique` — full scan `GROUP BY ... HAVING count(*)>1`, а не проба на сэмпле.**
`tools/toolbox.py:111-123`. DESIGN §8 прямо требует «живая проба на СЭМПЛЕ, не full scan»,
риск №3 реестра рисков. По факту — полная агрегация по таблице. На проде либо (а) упрётся
в `probe_max_cost` → `GuardError`, который **не перехватывается** в `join_advisor`
(`nodes.py:269` — `check_join` зовётся без try/except) → весь `ask()` падает исключением,
либо (б) при щедром потолке — минуты ожидания на каждый join × 2 стороны × перепроверки.
Любой вопрос с join к крупной прод-таблице сейчас или падает, или тормозит.

**B-3.** *(аналитические функции)* Полная загрузка таблицы в pandas без гейта —
перенесено в `ANALYTICS_CORE_DESIGN.md`, приложение (решается DataFrame Store с гейтом
по памяти).

**B-4. GigaChat-бэкенд игнорирует `max_tokens`.**
`llm/gigachat.py:67-88` — параметр принят и выброшен: в `GigaChat(...)` и `llm.invoke(messages)`
бюджет токенов не передаётся. Все тщательно подобранные бюджеты узлов на проде не действуют.
Длинный ответ обрежется дефолтом модели — и JSON-парсинг начнёт флакать именно там, где
его лечили.

**B-5. Worst-case каскад ретраев: до ~50 LLM-вызовов и ~10+ минут на один `complete_json`.**
`llm/client.py`: `complete_json` = 3 попытки × (генерация + ремонт), каждая из которых —
`complete` с лестницей из 8 попыток (4 быстрых + 3 медленных по 30 с). Под GigaChat каждый
вызов ещё ждёт глобальные 5 с (`MIN_INTERVAL`) и имеет **свои** 5 ретраев внутри бэкенда.
Слои ретраев перемножаются. Нужен единый бюджет ретраев на запрос, а не три независимых слоя.

### 3.2 Серьёзные (неверное поведение / тихая деградация)

**B-6. SQL-guard режет валидные запросы и основан на подстроках.**
`db/adapter.py:37-42, 182-189`: `_FORBIDDEN` ищет слова `set|do|call|comment|copy|merge|begin...`
как `\b`-токены **в любом месте строки, включая строковые литералы**. Литерал
`'%обновление (update)%'` — отклоняется; `;` внутри строкового литерала → ложный отказ
«несколько операторов». Raw-SQL-режим (`plan_build` → `raw_sql`) делает guard основной
линией обороны — а он дырявый в обе стороны: `SELECT pg_sleep(600)` или тяжёлый cross join
проходят (read-only ≠ безопасно), а безобидный литерал — нет.

**B-7. Выражения в плане рендерятся вербатим — контракт «SQL не уедет от плана» неполный.**
`plan/render.py:22-27, 40-47`: всё, что `_EXPR_RE` посчитал «выражением» (скобка, пробел,
`/`, `*`), вставляется в SQL как есть — включая подзапросы `(SELECT ...)`, которые guard
пропустит. Модель может (не по злому умыслу, а по глюку) положить в `projections.column`
коррелированный подзапрос, и он выполнится. Раз уж выражения разрешены — их надо парсить
(sqlglot) и валидировать по AST-белому списку функций, а не по regex.

**B-8. `resolve_values` ослепляет фильтры: `=` всегда превращается в `ILIKE '%...%'`.**
`nodes.py:204-209`: если значение нашлось пробой, оператор принудительно становится `ILIKE`
с обрамлением `%…%`. Для точного enum-значения (`status = 'Закрыта'`) это меняет семантику:
`'%Закрыта%'` зацепит «Не закрыта» — классический баг подстроки. `_best_match` «самое
короткое совпадение» — та же ловушка наоборот.

**B-9. Комментарии-описания сидируются по имени колонки глобально, без привязки к таблице.**
`catalog/refresh.py:152-174` (`_merge_desc_seed_from_csv`, `_col_seed` ключуется
`name.lower()`): колонка `status` из таблицы задач получит описание колонки `status` из
любой другой таблицы. Плюс **самоподдерживающийся цикл**: LLM-сгенерированное описание при
следующем рефреше становится «кураторским сидом» с приоритетом над свежим LLM — ошибка,
однажды сгенерированная, фиксируется навсегда.

**B-10. `_find_pk` — комбинаторный взрыв на широких таблицах.**
`catalog/refresh.py:77-96`: перебор `combinations(cols, size)` до size=4. На таблице в
60–100 колонок это миллионы комбинаций × `df.duplicated()` по 100k-сэмплу каждая — рефреш
зависнет без всякого таймаута (per-table timeout защищает только SQL-сэмпл, не клиентский
перебор).

**B-11. Overlap-пробы рефреша тихо деградируют join-граф.**
`catalog/refresh.py:496-509`: `_overlap` при любой ошибке (включая cost-потолок на больших
таблицах) возвращает `0` → пара выбрасывается. На проде join-граф может молча получиться
пустым, и вся анти-fanout-механика (`suggest_join`) деградирует в «нет кандидатов», о чём
никто не узнает.

**B-12. `GreenplumAdapter` — пустой класс, а различия реальны.**
`db/adapter.py:343-345`. `ORDER BY random()` на GP-таблице в миллиард строк, cost-модель GP
(другие масштабы Total Cost → один `probe_max_cost` на оба диалекта бессмыслен),
`DISTINCT ON` поверх distribution key, структура `EXPLAIN (FORMAT JSON)` со slice/motion
отличается — парсинг `data[0]["Plan"]["Total Cost"]` может не совпасть. «Диалект-граница»
задекларирована, но не реализована.

**B-13. Трассировка пишет полные промпты и данные на диск без ограничений.**
`trace.py` (все события с `system`/`user`/`text`), `logging_setup.py` (DEBUG-лог всего),
`nodes.py:400` (первые строки результата в `agent.log`). Для банковских данных (ИНН, ФИО,
ЗП) это несанкционированное копирование чувствительных данных в файлы `traces/*.jsonl` и
`workspace/agent.log`, которые README предлагает «прислать при сбое». Нужна
редакция/маскирование и ротация (сейчас `FileHandler` без ротации — лог растёт бесконечно).

**B-14. `root.handlers.clear()` уничтожает логирование хост-приложения.**
`logging_setup.py:28`. Библиотека, сносящая root-логгер процесса, в Jupyter с другими
расширениями — источник загадочных «у меня пропали логи».

**B-25.** *(аналитические функции)* Мутация df и глобального matplotlib при импорте —

## 4. Архитектурные слабости по подсистемам

### 4.1 Text2SQL-агент

1. **Граф выродился относительно DESIGN.** Задумка «короткий контекст, много мелких шагов»
   (`scope → explore → grain&agg → join_advisor → plan → validate`) на деле схлопнулась:
   узлов `explore` и `grain&agg` нет, `validate` растворён в `synth`. Вместо этого —
   **один гигантский PLAN_SYS на ~70 строк правил** (`graph/prompts.py:41-108`), где
   смешаны: анти-fanout, dedup, полуоткрытые интервалы, ILIKE-резолв, HAVING, выражения,
   окна, raw-SQL. Это прямо противоречит собственному принципу проекта и является худшим
   режимом для слабой модели: один вызов должен удержать в голове 15 правил сразу.
   Половина последующих узлов (`_correct_count_ids`, `resolve_values`, `join_advisor`) —
   это фактически **заплатки, исправляющие ошибки перегруженного плана**, вместо того
   чтобы не дать их совершить.

2. **Нет диалога.** `Agent.ask` открывает новый thread (`agent.py:121-126`), контекст
   предыдущего вопроса теряется. «Теперь добавь разрез по сегменту» после выполненного
   запроса — начинается с нуля, включая повторные пробы БД. Follow-up — базовое ожидание
   пользователя от такого инструмента.

3. **Нет накопления знаний.** Каждая коррекция пользователя («это поле не деньги», «джойни
   по old_gosb_id») исчезает после сессии. Одобренные (человеком!) пары вопрос→план — это
   золотой датасет few-shots, который никто не собирает, хотя трейсы уже пишутся.

4. **Коррекция перезапускает весь конвейер.** `present → plan_build` с добавленной строкой
   коррекции — план строится заново целиком. Заявленное в DESIGN «точечный re-run
   затронутого узла» не реализовано. Для мелкой правки («LIMIT 10») это лишние 2–4
   LLM-вызова и повторные пробы.

5. **`MemorySaver`** — состояние живёт в памяти ядра; перезапуск ноутбука убивает сессию.
   Персистентный checkpointer (файловый, без SQLite — см. решение владельца о
   JSON-хранилищах) стоил бы немного.

6. **Ambiguity-опции нельзя комбинировать** и нельзя ответить свободным текстом с
   уточнением — `_resolve_ambiguity` понимает только индекс/подстроку label.

7. **`recency_col` и авто-dedup содержат доменные русизмы** («карточк», «созда») —
   `toolbox.py:204-218`. Мелочь, но это ровно тот «хардкод под таблицы», который DESIGN
   запрещает.

### 4.2 LLM-слой

1. **Не используется function calling GigaChat.** GigaChat-3 поддерживает
   функции/структурные ответы через `langchain_gigachat` (`bind_tools` /
   `with_structured_output`). Вместо этого — «Ответь ТОЛЬКО валидным JSON…» +
   четырёхступенчатый самодельный ремонт (`_clean_json`, `_extract` со счётчиком скобок).
   Это главный источник хрупкости и лишних токенов. Со структурным выводом исчезает целый
   класс ретраев.

2. **Один системный + один user-месседж на вызов** — интерфейс `chat(system, user)` не
   позволяет ни few-shots как отдельных сообщений, ни продолжения диалога, ни префилла.
   Для слабой модели few-shot-примеры в роли assistant заметно стабильнее «схемы ответа
   в тексте».

3. **Rate-limit 5 с — константа класса**, не конфиг; usage/токены не учитываются
   (`LLMResult.usage` пуст для GigaChat) — стоимость и деградацию нельзя мониторить.

4. **Ретраи слоятся** (см. B-5) и `sleep` держится внутри глобального лока
   (`gigachat.py:61-65`) — при будущей параллелизации это станет узким местом.

### 4.3 DB-слой

1. Guard должен стать **парсером, а не regex** (sqlglot: один `parse_one`, проверка типа
   statement, обход AST на запрещённые узлы, автодобавление LIMIT через AST — и сразу
   решаются B-6, B-7, `_apply_limit`-обёртка и подсчёт «одного оператора»).
2. `run_select` делает **EXPLAIN перед каждой пробой** — два раундтрипа; на GP EXPLAIN
   сам по себе не бесплатен. Кэш планов/costs по нормализованному SQL напрашивается.
3. `estimate_row_count`, `metadata_sample` есть — но не используются нигде, кроме рефреша
   (аналитическим функциям они нужны для гейта по памяти — см. `ANALYTICS_CORE_DESIGN.md` §5.2).
4. Пул соединений один, `statement_timeout` один на всё (B-1): нужно два профиля
   соединений — probe (короткий таймаут) и export.

### 4.4 Каталог и метаданные

1. **CSV как хранилище** уже трещит: `|`-разделители (B-21), полная перезапись при каждом
   рефреше, никакого версионирования/истории. Целевое хранилище — JSON-файлы с атомарной
   записью (решение владельца; см. `ANALYTICS_CORE_DESIGN.md` §3.3), что заодно даёт
   инкрементальный рефреш и историю в git.
2. **PK-гипотеза** ищется перебором на сэмпле (B-10) — вместо этого стоит спрашивать
   статистику БД (`pg_stats.n_distinct`, GP: `gp_distribution_policy` — distribution key
   почти всегда входит в бизнес-ключ) и только потом добирать перебором по кандидатам
   с высокой уникальностью.
3. **Семантическая классификация** (`_classify`) — регекс по имени. LLM уже вызывается для
   описаний — та же итерация может выдавать и класс, и грануляцию, и роль таблицы, и связи
   (обогащённый каталог за один батч-вызов на таблицу; это и делает семантическая модель
   из `ANALYTICS_CORE_DESIGN.md` §4).
4. **Синтетические сэмплы генерируются на проде по умолчанию** (`gen_samples = bool(llm)`) —
   лишние тысячи токенов и минуты на каждую таблицу рефреша ради файла, нужного только
   для тестового контейнера.

### 4.5 CLI/UX

1. **Jupyter-REPL с блокирующим `input()`** — тупик: нет истории, нет автодополнения
   таблиц, нет прогресса LLM-вызовов (только `…статус`), невозможно прервать долгий шаг,
   вывод плана — plain text. Даже оставаясь в ноутбуке, `ipywidgets`-чат дал бы кликабельные
   варианты неоднозначности, кнопку «выполнить», рендер плана таблицей. Следующая ступень —
   тонкий web-UI (см. §6.2).
2. Парсинг команд — цепочка `elif` на 60 строк; `--where` вычленяется вручную. Мини-роутер
   команд с декларацией аргументов упростил бы и help, и валидацию.
3. Результат показывается как `df.head(20)` без итогов/форматирования чисел; SQL — после
   результата, а не до.

### 4.6–4.7 Бизнес-отчёт и расследования

Перенесено в `ANALYTICS_CORE_DESIGN.md`: диагноз качества — §1, перенесённые находки
аудита с привязкой к решениям — приложение (§13).

### 4.8 Eval и инженерная культура

1. 7 SQL-кейсов, ни одного негативного (инъекции/guard), ни одного на dedup-корректность
   значений (сумма после dedup = эталон), ни одного на «отказ» (вопрос вне данных).
   Eval аналитических функций — см. `ANALYTICS_CORE_DESIGN.md` §10.
2. Нет юнит-тестов на чистые функции, которых много и которые идеально тестируемы:
   `_normalize_on`, `normalize_plan`, `render_sql`, `_apply_limit`, `_find_pk`, guard.
   Это дешёвые сотни ассертов, ловящие регрессии без LLM.
3. Нет фиксации LLM-ответов (VCR-кассеты/golden-файлы) — eval недетерминирован и требует
   живой модели даже для проверки кода вокруг неё.

## 5. Где решения неуникальны (и что мир делает лучше)

| Место в харнесе | Самодельное решение | State of the art |
|---|---|---|
| SQL-guard, авто-LIMIT | regex-подстроки | **sqlglot**: AST-валидация, тип стейтмента, трансформации (LIMIT, квотинг), транспиляция Postgres↔Greenplum |
| JSON от LLM | «ответь только JSON» + ручной ремонт | **function calling / structured output GigaChat**, pydantic-схема как контракт |
| Retrieval таблиц | BM25 без морфологии | гибрид: эмбеддинги (мультиязычные bge-m3/e5 локально) + BM25 со стеммингом + rerank; для <50 таблиц хватит стемминг+синонимы из каталога |
| Выбор колонок/значений | весь стол колонок в один промпт | **schema linking** как отдельный шаг (выбор релевантных колонок до планирования) — стандарт всех топ-решений BIRD/Spider |
| Генерация SQL | один план с первого раза + заплатки | **multi-candidate + execution-based selection**: 2–3 кандидата (структурный план и raw-SQL), прогон EXPLAIN/пробы, самосогласованность результатов |
| Ошибки исполнения | goto error-конец | **self-correction loop**: текст ошибки БД → обратно модели с планом → повторный синтез (1–2 итерации). Есть только для JSON и fanout, но не для SQL-ошибок |
| Память | нет | хранение одобренных пар Q→план как few-shots с retrieval по похожести вопроса (self-improving loop без дообучения — то, что нужно при фиксированной gigachat-3-ultra) |
| Оценка | скрипт с print | pytest + golden-кассеты + матрица pass-rate в CI |

Неуникальные решения аналитических функций (semantic layer vs регексы «денег», Plotly +
Tabulator vs статические PNG, детекторы на агрегатах) — разобраны в
`ANALYTICS_CORE_DESIGN.md` (§5–6 и приложение).

## 6. Как сделать в разы лучше: целевая S-tier архитектура агента

Ограничение одно — `gigachat-3-ultra`. Значит выигрывать нужно не «моделью поумнее», а
**структурой вокруг модели**: меньше решений на вызов, структурный вывод, исполнение как
арбитр, память как усилитель. Целевая архитектура аналитических функций (дэши,
расследования, impact) — в `ANALYTICS_CORE_DESIGN.md`; ниже — агентная половина.
Семантическая модель из того же документа (§4) — общий фундамент: агент становится её
потребителем (проверенные join-пути, единицы, скоуп-правила).

### 6.1 Text2SQL: конвейер «мало решений на вызов» + исполнение как арбитр

```
Q → context builder (retrieval: таблицы + релевантные КОЛОНКИ + похожие прошлые Q→план)
  → schema linking (LLM-вызов №1: выбрать колонки/фильтры/интерпретации; surfaced ambiguity)
  → plan synthesis (LLM-вызов №2: StructuredPlan НА УЖЕ ОТОБРАННЫХ колонках; короткий промпт)
  → deterministic hardening (resolve_values c сохранением '=', join из семантической модели,
    normalize, sqlglot-валидация, EXPLAIN)
  → cheap execution probe (LIMIT 100) → если ошибка/пусто → self-correction (LLM-вызов №3
    с текстом ошибки; максимум 2 итерации)
  → present (план + превью 5 строк результата!) → ok → полный экспорт
```

Ключевые отличия от текущего:
1. **Schema linking отдельно от плана** — режет главный промпт вдвое и убирает заплатки
   `_correct_count_ids` (выбор идентификатора — это schema linking).
2. **Превью результата до апрува.** Сейчас пользователь окает SQL вслепую. Проба LIMIT 100
   стоит копейки и ловит 80% семантических ошибок («пусто», «дубли», «не те единицы») ДО
   полного исполнения. Самое дешёвое улучшение качества во всей системе.
3. **Self-correction по ошибкам БД** — сейчас любой EXPLAIN-fail = конец сессии.
4. **Память**: одобренный план сохраняется `{question, plan, tables}` (JSON, append-only);
   при новом вопросе top-2 похожих подмешиваются few-shots. С фиксированной моделью это
   единственный механизм «система становится умнее со временем» — и он почти бесплатный
   (трейсы уже есть).
5. **Диалог**: thread не сбрасывается; предыдущий план в state; «а теперь по месяцам» —
   это маленький LLM-вызов «модифицируй план», а не полный цикл.
6. Для сложных запросов — **два кандидата** (структурный план + raw-SQL) с выбором по
   исполнению на пробе: у слабой модели self-consistency даёт больший прирост, чем любой
   промпт-инжиниринг.

### 6.2 Платформа

1. **LLM-слой**: перейти на структурный вывод GigaChat (`with_structured_output`/functions);
   единый бюджет ретраев; passthrough `max_tokens`; учёт usage; конфигурируемый rate-limit;
   кэш вызовов по хэшу (system,user) — рефреш метаданных и повторные запуски станут почти
   бесплатными.
2. **DB-слой**: sqlglot-guard; два профиля соединений (probe 15 с / export 600 с — починить
   B-1); `key_unique` на сэмпле (`WHERE random()<p` подзапрос) с эскалацией до полного скана
   только по требованию; реальный `GreenplumAdapter` (cost-нормировка, distribution keys
   в join-советнике).
3. **Хранилище**: каталог + семантическая модель + память планов — **JSON-файлы с атомарной
   записью** (решение владельца: без SQLite — его файловые локи виснут из-за Jupyter-ядер;
   см. `ANALYTICS_CORE_DESIGN.md` §3.3); trace с маскированием значений (хэш вместо сырых
   данных) и ротацией.
4. **Пакетирование**: `pyproject.toml`, `pip install -e .`, extras `[prod]`/`[dev]`; убрать
   `sys.path.insert`; вычистить весь мёртвый код (§B-15); CI: ruff + mypy + pytest (юниты
   без LLM) + nightly eval с кассетами.
5. **UI**: ближний шаг — ipywidgets-чат в ноутбуке (кликабельные опции, кнопки ok/править,
   прогресс шагов); целевой — тонкий web-чат с историей сессий, живым стримом статусов
   графа (LangGraph это отдаёт событиями), таблицей результата и кнопкой «дэш по этому
   результату» — смычка с аналитическими поверхностями.

### 6.3 Eval как двигатель (а не приложение)

1. Юнит-слой: чистые функции (см. §4.8) — сотни быстрых тестов.
2. Кассеты LLM-ответов для детерминированного прогона конвейера без модели.
3. Расширить кейсы: негативные (guard, инъекции в фильтры), числовая корректность
   dedup-join, отказы («данных нет»), диалоговые (follow-up).
4. Метрики в CI: pass-rate по кейсам, латентность на кейс, число LLM-вызовов на кейс
   (регрессия эффективности — тоже регрессия).

## 7. Дорожная карта (порядок владельца: сначала аналитика, потом text2sql)

Зафиксированный порядок реализации: **первая очередь — аналитическое ядро**
(`ANALYTICS_CORE_DESIGN.md` §11), **вторая — перестройка text2sql**. Агент в первой
очереди продолжает работать как есть. Но часть агентных фиксов — общий фундамент,
без которого аналитическое ядро строить нельзя; они выносятся в «очередь 0».

### Очередь 0 — общий фундамент (дни; разблокирует аналитическое ядро)
1. **B-4** (max_tokens GigaChat) + **B-5** (единый бюджет ретраев) + переход на
   **структурный вывод GigaChat** — LLM-слой один на все поверхности: enrich,
   планировщик, критик и нарратор аналитического ядра строятся уже на нём.
2. **B-1** (таймауты проб: отдельный probe-профиль соединений) — от проб зависит
   overlap join-графа в `/refresh_metadata`, а от него — join-пути семантической модели.
3. **B-22** → общая утилита `atomic_write_json` — на ней строится всё JSON-хранилище
   ядра (semantics/, playbooks/, feedback/, golden/).
4. Удалить мёртвый код (**B-15**, обе части), добавить `pyproject.toml`, первые
   юнит-тесты на render/model/guard (**B-26**); попутно **B-14** (не сносить
   root-хендлеры логирования).

### Очередь 1 — аналитическое ядро
Реализуется по `ANALYTICS_CORE_DESIGN.md` §11 (фазы 1–4: семантическая модель →
engine/render → /dash → примитивы+плейбуки → смычка и фидбек). Побочно закрываются
каталожные находки **B-9, B-10, B-11** — рефреш метаданных поглощается enrich-конвейером
семантической модели, и **B-21** — CSV-каталог уступает JSON.

### Очередь 2 — перестройка text2sql (после ядра)

Быстрые:
1. **B-2** (key_unique на сэмпле + try/except в join_advisor);
2. **B-8** (сохранять `=` при точном совпадении значения); **B-20** (пустой ввод ≠ ok).

Средние:
3. sqlglot: guard, LIMIT, валидация выражений плана, транспиляция диалекта (**B-6, B-7**);
4. превью результата (LIMIT 100) перед апрувом + self-correction по ошибкам БД;
5. schema linking как отдельный узел; сокращение PLAN_SYS вдвое; нормализация op/agg
   (**B-19**);
6. диалоговый режим (не сбрасывать thread) + память одобренных планов как few-shots;
7. агент переходит с CSV-каталога на семантическую модель ядра (join-пути/единицы/скоуп);
   BM25 со стеммингом (**B-23**); реальный `GreenplumAdapter` (**B-12**); маскирование
   и ротация трейсов (**B-13**).

Долгие:
8. multi-candidate генерация с execution-based выбором;
9. web-чат UI со стримом шагов графа, историей и «дэш по результату» (смычка с ядром);
10. nightly-eval в CI с трендом pass-rate (**B-24** — детерминированный выбор опции).

### Финальное слово

Фундамент выбран верно: инварианты вместо «магии», человек в контуре, детерминированные
числа, LLM только там, где нужен смысл. Именно поэтому у агента есть реальный путь к
S-tier: не переписывание идеи, а (1) выправление прод-рисков B-1…B-5, (2) переход
guard/рендера на AST, (3) исполнение как арбитр качества (превью + self-correction +
кандидаты), (4) память и диалог как компенсатор фиксированной модели и (5) общий фундамент
с аналитическим ядром (`ANALYTICS_CORE_DESIGN.md`). Каждый шаг усиливает остальные — и все
они достижимы на gigachat-3-ultra, потому что уменьшают, а не увеличивают требования к
интеллекту модели.
