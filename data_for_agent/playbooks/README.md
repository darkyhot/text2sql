# Плейбуки расследований — как добавить новый сложный дэш/расследование

Плейбук — это **JSON-рецепт** (не код): цепочка проверенных примитивов над данными, которую
исполняет один executor. Новый класс вопросов для директора = новый JSON-файл здесь, а не новая
команда и не новый модуль. LLM (FRAME) только подставляет параметры; числа считают примитивы
детерминированно (pandas).

Механика: `plans.py` (схема + executor `run_plan`), `primitives.py` (библиотека операций),
`surface.py` (bindings + рендер шагов в ECharts), роутинг — в `investigate._route_playbook`.

## Схема рецепта

```jsonc
{
  "name": "benchmark",                       // имя = имя файла
  "description": "Кто лучше/хуже и почему",   // человекочитаемо
  "when": {                                  // условия срабатывания (роутинг)
    "question_patterns": ["кто лучше", "бенчмарк", "сравни отделения", "хуже среднего"],
    "requires_flag": false                   // true → нужен tool_flag в семмодели (как impact)
  },
  "params": ["target", "dim", "entity"],     // какие параметры рецепт ждёт
  "plan": {
    "question": "Кто отклоняется от медианы и почему",
    "steps": [
      {"id": "scale", "primitive": "aggregate",  "params": {"measure": "$target", "agg": "sum"}},
      {"id": "rank",  "primitive": "decompose",  "params": {"measure": "$target", "dim": "$dim", "side": "top"}},
      {"id": "who",   "primitive": "pareto",     "params": {"measure": "$target", "entity": "$entity", "side": "top"}}
    ],
    "verdict_template": "Кто в топе ({rank}) → концентрация ({who})"
  }
}
```

### Как связываются шаги (params)
- `"$name"` — параметр плейбука из bindings (заполняет FRAME/эвристика).
- `"$stepid.facts.key"` — факт предыдущего шага (напр. `"$did.facts.did"` в impact).
- литерал (число/строка `"loss"`) — как есть.
- Если ссылка не разрешилась (нет колонки/факта на этой таблице) — шаг **честно пропускается**,
  расследование не падает.

## Библиотека примитивов (то, из чего собираются рецепты)

| Примитив | Что делает |
|---|---|
| `aggregate` | мера × разрезы × фильтры (сумма/среднее/…); базовый агрегат |
| `decompose` | вклад срезов в итог/потери (топ-N + «прочие») |
| `pareto` | концентрация по сущностям: топ-N = X% |
| `trend_break` | динамика + детекция перелома, до/после точки |
| `lift_profile` | «почему»: доля причины в срезе против базы (lift ×N) |
| `compare_groups` | наивное сравнение A vs B (с явной пометкой) |
| `did` | difference-in-differences: Δ(пилот) − Δ(контроль) |
| `stratified_compare` | сравнение с/без внутри однородных страт (взвешенный эффект) |
| `selection_check` | проверка самоотбора: исходы будущих пилотов в до-периоде |
| `placebo_check` | эффект на мере, где его быть не должно (≈0 = честно) |
| `extrapolate` | перенос эффекта на неохваченный объём, с допущениями |

Полный актуальный список — `python -c "from src.text2sql.report import primitives; print(primitives.names())"`.

## Как добавить свой плейбук — три случая

**1. Частый случай: рецепт из существующих примитивов и параметров → 0 кода.**
Имена параметров `target/dim/entity/reason/date/side` (loss-семейство) и
`flag/outcome/time_col/cutoff/strata/placebo` (impact-семейство) FRAME уже умеет привязывать.
Достаточно:
1. положить `<name>.json` сюда;
2. задать `when.question_patterns` (по ним роутится);
3. готово — `/investigate <таблица> <вопрос>` подхватит рецепт автоматически.

**2. Нужны НОВЫЕ имена параметров → +пара строк.**
Добавь их в промпт FRAME (`surface._FRAME_SYS`) и в эвристический фолбэк (`surface._bindings`),
чтобы FRAME знал, какие колонки в них класть. Это не новый модуль — правка в одном месте.

**3. Нужна НОВАЯ математика, которой нет среди примитивов → +один примитив с тестом.**
Добавь функцию с декоратором `@primitive` в `primitives.py` (вход/выход — `PrimitiveResult`
с `facts`) и юнит-тест на эталонных числах в `eval/`. Только этот случай требует кода —
и только математики, не команд и не рендера.

## Что готово, а что расширение (§8 дизайна)
- **Готово:** библиотека из 11 примитивов, «план как данные» + executor, JSON-рецепты, роутинг в
  `/investigate`, рендер шагов в ECharts, вердикт. Рецепты полки: `impact`, `loss_attribution`.
- **Расширение (JSON-ом, без переписывания ядра):** `benchmark`, `funnel_drop`, `churn_risk` и
  ad-hoc путь (LLM собирает план из примитивов, когда готового рецепта нет).

## Проверка нового рецепта
```
/investigate <схема.таблица> <вопрос, попадающий в question_patterns>
```
В прогрессе будет строка «вопрос распознан как «<name>» — исполняю плейбук…». Результат:
`workspace/<таблица>_playbook.html` (+ `.md`). Пропущенные шаги перечислены в футере —
это норма, если часть параметров неприменима к таблице.
